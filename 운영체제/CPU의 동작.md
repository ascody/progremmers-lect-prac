# CPU는 메모리의 값을 어떻게 레지스터에 적재해서 동작할까?

## 1. 핵심 요약

CPU는 메모리에 있는 값을 직접 계산하지 않는다.

대신 다음과 같은 흐름으로 동작한다.

```text

메모리에서 값을 가져온다

→ 레지스터에 적재한다

→ 레지스터 안의 값으로 연산한다

→ 필요하면 결과를 다시 메모리에 저장한다

```

즉, CPU 연산의 중심은 **메모리**가 아니라 **레지스터**이다.

---

## 2. 메모리와 레지스터의 차이

### 메모리 RAM

메모리는 프로그램 코드와 데이터를 저장하는 공간이다.

예를 들어 다음과 같은 값들이 메모리에 저장될 수 있다.

```text

주소 1000: b = 3

주소 1004: c = 5

주소 1008: a = ?

```

메모리는 많은 데이터를 저장할 수 있지만, CPU 내부 저장공간에 비해 상대적으로 느리다.

### 레지스터 Register

레지스터는 CPU 내부에 있는 아주 작은 저장공간이다.

용량은 작지만 CPU가 바로 접근할 수 있어서 매우 빠르다.

CPU는 보통 메모리에 있는 값을 레지스터로 가져온 뒤, 레지스터 값을 이용해 연산한다.

---

## 3. 예시: `a = b + c`

고급 언어에서는 다음과 같이 작성할 수 있다.

```c

int a = b + c;

```

하지만 CPU 내부에서는 대략 다음과 같은 흐름으로 실행된다.

```assembly

LOAD R1, [1000]   ; 메모리 주소 1000의 값을 R1 레지스터에 적재

LOAD R2, [1004]   ; 메모리 주소 1004의 값을 R2 레지스터에 적재

ADD  R3, R1, R2   ; R1과 R2를 더해서 R3에 저장

STORE [1008], R3  ; R3 값을 메모리 주소 1008에 저장

```

흐름을 그림처럼 보면 다음과 같다.

```text

메모리[1000] = 3  ──LOAD──>  R1 = 3

메모리[1004] = 5  ──LOAD──>  R2 = 5

  

R1 + R2 ──ADD──> R3 = 8

  

R3 = 8 ──STORE──> 메모리[1008] = 8

```

즉, CPU는 `b + c`를 메모리에서 직접 계산하지 않는다.

메모리에서 값을 가져와 레지스터에 넣고, 레지스터 안에서 계산한 뒤, 결과를 다시 메모리에 저장한다.

---

## 4. CPU의 기본 실행 흐름

CPU는 명령어를 보통 다음 세 단계로 처리한다.

```text

Fetch → Decode → Execute

```

---

## 5. Fetch: 명령어 가져오기

CPU는 먼저 메모리에서 다음에 실행할 명령어를 가져온다.

이때 CPU 내부에는 **PC, Program Counter**라는 레지스터가 있다.

PC는 다음에 실행할 명령어의 메모리 주소를 저장한다.

예를 들어 다음과 같은 상태라고 하자.

```text

PC = 2000

메모리[2000] = LOAD R1, [1000]

```

CPU는 PC가 가리키는 주소인 `2000`번지에서 명령어를 가져온다.

```assembly

LOAD R1, [1000]

```

이 과정이 **Fetch**이다.

---

## 6. Decode: 명령어 해석하기

CPU는 가져온 명령어를 해석한다.

```assembly

LOAD R1, [1000]

```

이 명령어를 CPU는 다음과 같이 이해한다.

```text

메모리 주소 1000에 있는 값을 읽어서

R1 레지스터에 넣으라는 뜻이구나.

```

이 과정이 **Decode**이다.

---

## 7. Execute: 명령어 실행하기

이제 CPU는 실제로 명령어를 실행한다.

```text

메모리[1000] = 3

```

CPU는 메모리 주소 `1000`의 값을 읽고, 그 값을 R1 레지스터에 넣는다.

```text

R1 = 3

```

이 과정이 **Execute**이다.

그 후 PC는 다음 명령어 주소로 이동한다.

```text

PC = 2004

```

그리고 CPU는 다시 Fetch 단계부터 반복한다.

---

## 8. CPU는 변수 이름이 아니라 주소로 접근한다

프로그래밍 언어에서는 다음처럼 변수 이름을 사용한다.

```c

int a = b + c;

```

하지만 CPU는 `a`, `b`, `c` 같은 변수 이름을 직접 이해하지 않는다.

컴파일 이후에는 변수들이 메모리 주소로 바뀐다.

```text

b → 메모리 주소 1000

c → 메모리 주소 1004

a → 메모리 주소 1008

```

그래서 CPU는 다음과 같이 주소를 기준으로 값을 가져온다.

```assembly

LOAD R1, [1000]

LOAD R2, [1004]

```

정리하면, CPU는 변수를 이름으로 찾는 것이 아니라 **메모리 주소를 통해 값에 접근**한다.

---

## 9. 실제로는 캐시가 중간에 있다

단순하게 설명하면 CPU가 메모리에서 값을 가져온다고 말할 수 있다.

하지만 실제 컴퓨터 구조에서는 CPU와 RAM 사이에 **캐시 메모리**가 있다.

일반적인 계층 구조는 다음과 같다.

```text

CPU 레지스터

   ↑

L1 캐시

   ↑

L2 캐시

   ↑

L3 캐시

   ↑

RAM

```

CPU가 어떤 메모리 주소의 값을 요청하면 먼저 가까운 캐시부터 확인한다.

```text

1. L1 캐시에 값이 있는지 확인

2. 없으면 L2 캐시 확인

3. 없으면 L3 캐시 확인

4. 그래도 없으면 RAM에서 가져옴

```

즉, 우리가 “메모리에서 레지스터로 가져온다”고 말하더라도 실제로는 캐시 계층을 거칠 수 있다.

---

## 10. 전체 흐름 정리

CPU가 메모리 값을 레지스터에 적재해서 동작하는 과정은 다음과 같다.

```text

1. PC가 다음 명령어 주소를 가리킨다.

2. CPU가 메모리에서 명령어를 가져온다. Fetch

3. CPU가 명령어를 해석한다. Decode

4. LOAD 명령이면 특정 메모리 주소의 값을 읽는다.

5. 읽은 값을 CPU 내부 레지스터에 넣는다.

6. ADD, SUB 같은 연산은 레지스터 값을 이용해 수행한다.

7. 결과를 필요하면 STORE 명령으로 다시 메모리에 저장한다.

8. PC가 다음 명령어 주소로 이동한다.

9. 위 과정을 반복한다.

```

---

## 11. 한 문장으로 요약

CPU는 메모리에 있는 코드와 데이터를 주소로 가져오고, 데이터를 레지스터에 올린 뒤, 레지스터 기반으로 연산하고, 결과를 다시 메모리에 저장하면서 프로그램을 실행한다.

---

## 12. Java 코드로 다시 생각해보기

Java 코드에서는 다음처럼 보일 수 있다.

```java

int a = b + c;

```

하지만 CPU 내부에서는 대략 다음과 같은 흐름이다.

```text

b 값을 메모리에서 R1으로 가져온다.

c 값을 메모리에서 R2로 가져온다.

R1과 R2를 더해서 R3에 저장한다.

R3 값을 a가 저장될 메모리 위치에 저장한다.

```

즉, 프로그래밍 언어의 한 줄도 CPU 입장에서는 여러 개의 작은 명령어로 나뉘어 실행된다.

---

## 13. 핵심 키워드

| 용어 | 의미 |

|---|---|

| CPU | 명령어를 실행하고 연산을 수행하는 장치 |

| 메모리 RAM | 프로그램 코드와 데이터를 저장하는 공간 |

| 레지스터 | CPU 내부의 아주 빠른 임시 저장공간 |

| PC | 다음에 실행할 명령어의 주소를 저장하는 레지스터 |

| LOAD | 메모리의 값을 레지스터로 가져오는 명령 |

| STORE | 레지스터의 값을 메모리에 저장하는 명령 |

| ADD | 레지스터 값을 이용해 덧셈하는 명령 |

| Fetch | 명령어를 메모리에서 가져오는 단계 |

| Decode | 명령어를 해석하는 단계 |

| Execute | 명령어를 실제로 실행하는 단계 |

| Cache | CPU와 RAM 사이에 있는 빠른 임시 저장공간 |
