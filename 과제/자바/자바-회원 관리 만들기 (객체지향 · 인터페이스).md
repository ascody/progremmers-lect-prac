# 회원 관리 만들기 (객체지향 · 인터페이스)

> 회원 관리 프로그램을 **인터페이스(`interface`) + 구현 + 다형성**으로 설계합니다. 추상 클래스 버전과 기능은 같지만, 회원의 "규약"을 인터페이스 `Member`로 정합니다. **이 가이드는 "추상 클래스와 무엇이 다른지"를 직접 느끼는 비교 학습용**이기도 해요. **아래 Step을 순서대로 따라가면, 마지막 Step에서 프로그램이 완성됩니다.**

---

## 0. 먼저 알아둘 점 (중요)

이 회원 예제는 사실 **추상 클래스가 더 잘 맞습니다.** 회원들이 이름·이메일·연락처라는 **공통 상태**와 getter·printInfo 같은 **공통 구현**을 많이 공유하는데, 인터페이스는 상태(필드)를 가질 수 없어서 그 코드가 각 클래스에 **중복**되거든요.

그래서 이 가이드의 목표는 두 가지예요.

1. 인터페이스로 같은 프로그램을 구현해 보기
2. **왜 이 경우엔 추상 클래스가 나은지**, 그리고 `default` 메서드로 중복을 줄이는 법 배우기

---

## 1. 무엇을 만드나요?

요금제로 정원을 정하고, 회원(일반/VIP)을 추가·조회·수정·삭제합니다. 등급마다 혜택이 다릅니다.

```

[1]회원추가 [2]회원조회(메일) [3]회원조회(이름)

> 4
[VIP] 홍길동 / hong@a.com / 010-1111 (혜택: 10% 할인 + 무료배송)
[일반] 김철수 / kim@b.com / 010-2222 (혜택: 기본 서비스)
```

## 2. 핵심 개념

### (1) 인터페이스 = 규약 (메서드 목록)

인터페이스는 **필드(상태)도 생성자도** 가질 수 없습니다. "이런 메서드를 갖춰라"는 약속만 합니다. 이 인터페이스를 `implements` 하는 클래스는 약속한 메서드를 **모두** 구현해야 합니다.

### (2) `default` 메서드 (Java 8+)

인터페이스에서도 **다른 메서드만 이용하는 공통 동작**은 `default` 키워드로 구현을 넣을 수 있어요. `printInfo()`는 `getGrade()`, `getName()` 등만 쓰므로 인터페이스 안에 둘 수 있습니다.

### (3) 추상 클래스 vs 인터페이스 (이 예제 기준)

||추상 클래스 `Member`|인터페이스 `Member`|
|---|---|---|
|이름·이메일·연락처 필드|부모에 **한 번만**|각 클래스가 **직접** (중복!)|
|생성자|있음 (`super(...)`)|없음 → 각 클래스가 직접|
|getter / update|부모에 한 번 구현 → 물려받음|각 클래스가 **각자** 구현 (중복!)|
|printInfo|부모 메서드|`default` 메서드로 가능|
|다중 적용|단일 상속|여러 개 동시 구현 가능|

> 핵심: **공통 상태·구현이 많은 이 예제는 추상 클래스가 중복이 적어 유리**합니다. 인터페이스는 "여러 규약을 동시에 갖춰야 할 때"나 "상태 없는 순수 기능 약속"에 강합니다.

## 3. 파일 구조

|파일|역할|
|---|---|
|`Member.java`|**인터페이스**. 회원이 갖춰야 할 메서드 목록 (+ `default printInfo`)|

## 4. Step by Step

각 Step에 **목표 / 할 일 / 힌트 / 확인 방법**이 있습니다.

---

### Step 1. 회원 규약 — 인터페이스 (`Member.java`)

**목표**: 회원이 갖춰야 할 메서드를 인터페이스로 선언하고, 공통 출력은 `default`로 둔다.

**할 일**

1. `interface Member`.
2. `getName/getEmail/getPhone`, `getGrade`, `getBenefit`, `update`를 선언.
3. `printInfo()`는 `default` 메서드로 구현 (다른 메서드만 사용).

**힌트**

```java
public interface Member {
    String getName();
    String getEmail();
    String getPhone();
    String getGrade();
    String getBenefit();
    void update(String name, String email, String phone);

    // default 메서드: 인터페이스 안의 다른 메서드만 써서 구현 가능
    default void printInfo() {
        System.out.println("[" + getGrade() + "] " + getName() + " / " + getEmail()
                           + " / " + getPhone() + " (혜택: " + getBenefit() + ")");
    }
}
```

**확인**: 컴파일 에러가 없으면 OK.

---

### Step 2. 일반 회원 (`NormalMember.java`)

**목표**: `Member`를 구현한 일반 회원을 만든다. 인터페이스엔 필드가 없으니 **이름·이메일·연락처를 직접** 가진다.

**할 일**

1. `class NormalMember implements Member`.
2. 필드 `name`, `email`, `phone` + 생성자.
3. 모든 메서드 구현 (getter들, getGrade="일반", getBenefit, update).

**힌트**

```java
public class NormalMember implements Member {
    private String name, email, phone;

    public NormalMember(String name, String email, String phone) {
        this.name = name;
        this.email = email;
        this.phone = phone;
    }

    @Override public String getName()  { return name; }
    @Override public String getEmail() { return email; }
    @Override public String getPhone() { return phone; }

    @Override public String getGrade()   { return "일반"; }
    @Override public String getBenefit() { return "기본 서비스"; }

    @Override public void update(String name, String email, String phone) {
        this.name = name;
        this.email = email;
        this.phone = phone;
    }
}
```

**확인**: 일반 회원을 만들어 `printInfo()` 하면 `[일반] ...`로 나오면 성공.

---

### Step 3. VIP 회원 (`VipMember.java`) — 중복을 직접 느껴보기

**목표**: 같은 방식으로 VIP 회원을 만든다.

**할 일**: `NormalMember`를 본떠 **거의 똑같이** 만들고, `getGrade`/`getBenefit`만 바꾼다.

**힌트**

```java
public class VipMember implements Member {
    private String name, email, phone;     // ← NormalMember와 똑같음 (중복!)

    public VipMember(String name, String email, String phone) {  // ← 똑같음 (중복!)
        this.name = name; this.email = email; this.phone = phone;
    }

    @Override public String getName()  { return name; }   // ← 똑같음 (중복!)
    @Override public String getEmail() { return email; }
    @Override public String getPhone() { return phone; }

    @Override public String getGrade()   { return "VIP"; }              // 여기만 다름
    @Override public String getBenefit() { return "10% 할인 + 무료배송"; } // 여기만 다름

    @Override public void update(String name, String email, String phone) { // ← 똑같음 (중복!)
        this.name = name; this.email = email; this.phone = phone;
    }
}
```

> **느껴보기**: 필드·생성자·getter·update가 `NormalMember`와 완전히 똑같죠? 이게 바로 "인터페이스는 공통 구현을 물려줄 수 없어서 생기는 중복"입니다. **추상 클래스였다면 부모에 한 번만 쓰면 됐어요.**

**확인**: VIP의 `printInfo()`가 일반과 다르게 나오면 성공.

---

### Step 4. 회원 관리자 — 추가 (`MemberManager.java`)

**목표**: 회원 배열과 인원수를 관리하고, 회원을 추가한다. (추상 클래스 버전과 **동일**)

**힌트**

```java
public class MemberManager {
    private Member[] members;       // 인터페이스 타입 배열
    private int memberCnt;

    public MemberManager(int capacity) {
        members = new Member[capacity];
        memberCnt = 0;
    }

    public boolean isFull() { return memberCnt == members.length; }

    public boolean existsEmail(String email) {
        for (int i = 0; i < memberCnt; i++)
            if (members[i].getEmail().equals(email)) return true;
        return false;
    }

    public void add(Member m) {
        members[memberCnt] = m;
        memberCnt++;
    }

    public int getCount()    { return memberCnt; }
    public int getCapacity() { return members.length; }
}
```

> 사용하는 쪽(`MemberManager`)은 인터페이스든 추상 클래스든 똑같이 `Member`로 다룹니다.

**확인**: 매니저에 회원을 add 하면 `getCount()`가 늘면 성공.

---

### Step 5. 조회 (`MemberManager.java`)

**목표**: 이메일/이름으로 찾고, 전체를 출력한다.

**힌트**

```java
public Member findByEmail(String email) {
    for (int i = 0; i < memberCnt; i++)
        if (members[i].getEmail().equals(email)) return members[i];
    return null;
}

public Member findByName(String name) {
    for (int i = 0; i < memberCnt; i++)
        if (members[i].getName().equals(name)) return members[i];
    return null;
}

public void printAll() {
    for (int i = 0; i < memberCnt; i++) members[i].printInfo();  // default 메서드 호출
}
```

**확인**: 이메일/이름 조회, 전체조회가 동작하면 성공.

---

### Step 6. 수정 & 삭제 (`MemberManager.java`)

**힌트**

```java
public boolean update(String email, String name, String newEmail, String phone) {
    Member m = findByEmail(email);
    if (m == null) return false;
    m.update(name, newEmail, phone);
    return true;
}

public boolean delete(String email) {
    int idx = -1;
    for (int i = 0; i < memberCnt; i++)
        if (members[i].getEmail().equals(email)) { idx = i; break; }
    if (idx == -1) return false;

    for (int i = idx; i < memberCnt - 1; i++)
        members[i] = members[i + 1];   // 객체 참조 하나만 당기면 됨
    members[memberCnt - 1] = null;
    memberCnt--;
    return true;
}
```

**확인**: 수정 후 조회 시 바뀌고, 삭제 후 빈칸 없이 이어지면 성공.

### Step 7. 요금제 + 메뉴 루프 (`Main.java`)

**목표**: 요금제로 정원을 정하고, 메뉴를 반복하며 매니저 기능을 호출한다. 추가 시 등급을 골라 알맞은 회원을 만든다.

**힌트**

```java
Scanner sc = new Scanner(System.in);
System.out.println("[1]Lite:10 [2]Basic:20 [3]Premium:30");
int plan = Integer.parseInt(sc.nextLine());
MemberManager manager = new MemberManager(plan * 10);

// 회원추가 부분 (case 1)
if (manager.isFull()) {
    System.out.println("회원이 꽉 찼습니다.");
} else {
    System.out.println("등급 [1]일반 [2]VIP");
    int grade = Integer.parseInt(sc.nextLine());
    System.out.print("이름 > ");   String name  = sc.nextLine();
    System.out.print("이메일 > "); String email = sc.nextLine();
    System.out.print("연락처 > "); String phone = sc.nextLine();

    if (manager.existsEmail(email)) {
        System.out.println("이미 존재하는 회원입니다.");
    } else {
        Member m = (grade == 2) ? new VipMember(name, email, phone)
                                : new NormalMember(name, email, phone);
        manager.add(m);   // 어떤 등급이든 Member로 추가 (다형성)
    }
}
```

> **팁**: 입력은 `Integer.parseInt(sc.nextLine())`로 받고 `Scanner`는 하나만 공유하면 입력 꼬임을 피할 수 있어요.

**확인**: 등급을 골라 추가하고, 전체조회에서 등급·혜택이 다르게 보이면 성공!

## 5. 두 버전 비교로 마무리하기

학생들과 이렇게 정리하면 좋아요.

- **추상 클래스가 유리한 경우 (이 예제!)**: 공통 **상태(필드)** 와 공통 **구현**이 많을 때. → 회원의 이름·이메일·연락처·getter·update를 부모에 한 번만.
- **인터페이스가 유리한 경우**: 상태 없이 "이런 기능을 갖춰라"는 **규약만** 강제할 때, 또는 **여러 규약을 동시에** 적용해야 할 때 (`implements A, B`).

> 한 줄 요약: **상태·공통 구현을 물려주려면 추상 클래스, 규약만 강제하거나 다중 적용이 필요하면 인터페이스.**

## 6. (선택) 도전 과제

1. **다중 구현**: `Member`에 더해 `Comparable<Member>`를 함께 구현해 이름순 정렬 (인터페이스만의 장점!)
2. **중복 줄이기 실험**: 인터페이스 버전의 중복을 추상 클래스로 바꿔보고, 코드가 얼마나 줄어드는지 비교
3. **default 메서드 확장**: 등급별 인사말 같은 공통 동작을 `default`로 더 추가
4. **컬렉션으로**: `Member[]` 대신 `ArrayList<Member>` 사용
