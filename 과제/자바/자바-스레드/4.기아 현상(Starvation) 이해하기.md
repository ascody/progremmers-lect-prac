# 기아 현상(Starvation) 이해하기 (notify vs notifyAll)

> `wait()`/`notify()`를 배웠다면, 이번엔 **기아 현상(Starvation)** 과 `notifyAll()`을 익힙니다.
> 기아 현상은 특정 스레드가 **계속 자원을 못 얻어** 굶는 상황이에요.
> **먼저 `notify()`로 기아를 재현**하고, **`notifyAll()`로 해결**하는 흐름으로 진행합니다.
> **아래 Step을 순서대로 따라가면, "왜 notifyAll이 더 안전한지"를 눈으로 보게 됩니다.**
>
> 💡 각 Step의 **힌트는 접혀 있어요.** 먼저 스스로 해보고, 막히면 "힌트 보기"를 펼치세요.

---

## 0. 먼저 알아둘 점

이 과제는 `synchronized`와 `wait()`/`notify()`를 **이미 배웠다는 전제**예요.
스레드 결과는 비결정적이라 **기아 현상이 매번 똑같이 보이진 않을 수 있어요.** 어떤 일꾼이 굶는지, 얼마나 심한지는 실행마다 다릅니다. 우리는 "notify()에서는 특정 일꾼이 잘 못 얻고, notifyAll()에서는 고르게 얻는다"는 **경향**을 관찰하는 게 목표예요.

---

## 1. 무엇을 하나요?

여러 일꾼(worker) 스레드가 하나의 자원을 기다리고, 공급자가 주기적으로 자원을 내놓습니다.

```
worker1 is waiting for resource...
worker2 is waiting for resource...
worker3 is waiting for resource...
Resource is now available!
worker2 got the resource!        ← notify(): 특정 일꾼만 자꾸 얻고, 누군가는 계속 굶을 수 있음
...
(notifyAll로 바꾸면 → worker1, 2, 3이 고르게 얻음)
```

---

## 2. 학습 목표

| 개념 | 어디서 배우나 |
|------|------|
| 기아 현상(Starvation) | 특정 스레드가 계속 자원을 못 얻는 상황 |
| `notify()`의 한계 | 임의의 하나만 깨워서 누군가 굶을 수 있음 |
| `notifyAll()` | 모두 깨워 경쟁시켜 기아 완화 |
| 공유 자원 + 대기/공급 | wait()로 대기, notify(All)로 공급 알림 |

---

## 3. 핵심 개념

### (1) 기아 현상(Starvation)이란?
멀티스레드에서 **특정 스레드가 다른 스레드들에게 계속 밀려 자원을 얻지 못하고** 실행되지 못하는 상황이에요. 비유하면, 음식이 나올 때마다 매번 다른 사람들이 먼저 가져가서 **나만 계속 굶는** 상황이죠.

### (2) `notify()`는 왜 기아를 부를까?
`notify()`는 대기실의 스레드 중 **임의로 하나만** 깨워요. 그런데 어떤 스레드가 깨워질지는 보장이 없어서, **운 나쁜 스레드는 계속 안 깨워질 수** 있습니다. 그러면 그 스레드는 계속 굶어요.

### (3) `notifyAll()`로 완화
`notifyAll()`은 **대기 중인 모두를 깨웁니다.** 깨어난 스레드들이 다 같이 자원을 두고 경쟁해요. 비록 자원을 실제로 얻는 건 한 번에 하나지만, **모두가 매번 기회를 얻기** 때문에 시간이 지나면 고르게 분배됩니다.

> 한 줄 정리: **notify() = 한 명만 콕 깨움(누군가 굶을 위험), notifyAll() = 다 깨워서 공정하게 경쟁.**

### (4) 주의 — notifyAll도 "완벽한 공정"은 아님
notifyAll()도 결국 락을 얻는 건 하나뿐이고, 누가 얻을지는 스케줄러가 정해요. 그래서 **완벽한 순서 보장**은 아니지만, **기아를 크게 줄이는** 실용적 해법입니다. (완벽한 공정성은 fair lock 같은 도구로.)

---

## 4. 파일 구조

| 구성 | 역할 |
|------|------|
| `SharedResource` | 공유 자원. `isAvailable` + 대기(waitForResource) / 공급(makeResourceAvailable) |
| `WorkerThread` | 자원을 계속 요청하는 일꾼 |
| `main` | 일꾼 여러 명 + 주기적 공급자 스레드 실행 |

## 5. Step by Step

각 Step에 **목표 / 할 일 / 힌트(접힘) / 관찰 포인트**가 있습니다.

---

### Step 1. 공유 자원 골격 (`SharedResource`)

**목표**: 일꾼들이 공유하는 자원 객체와, 자원이 준비됐는지 나타내는 flag를 만든다.

**할 일**: `boolean isAvailable = false`.

<details>
<summary>💡 힌트 보기</summary>

```java
class SharedResource {
    private boolean isAvailable = false;   // 자원이 준비됐는가?
}
```

</details>

**확인**: 컴파일 에러가 없으면 OK.

---

### Step 2. 자원 대기 — waitForResource (`SharedResource`)

**목표**: 자원이 없으면 기다리고, 생기면 가져간다(소비).

**할 일**
1. `synchronized` 메서드.
2. `isAvailable`이 `false`면 대기 메시지 출력 후 `wait()`.
3. 자원을 얻으면 "got" 출력 → `isAvailable = false`.

<details>
<summary>💡 힌트 보기</summary>

```java
public synchronized void waitForResource(String threadName) {
    while (!isAvailable) {                 // 자원이 없으면
        try {
            System.out.println(threadName + " is waiting for resource...");
            wait();                         // 락 내려놓고 대기
        } catch (InterruptedException e) {
            throw new RuntimeException(e);
        }
    }
    System.out.println(threadName + " got the resource!");
    isAvailable = false;                    // 자원 소비
}
```

`while`로 감싸는 걸 잊지 마세요! (깨어난 뒤 조건 재확인)

</details>

**확인**: (실행은 Step 5) 자원이 없으면 대기, 생기면 가져가는 구조면 OK.

---

### Step 3. 자원 공급 — makeResourceAvailable (일단 notify로!)

**목표**: 자원을 공급하고 대기 중인 스레드에게 알린다. **문제를 보기 위해 먼저 `notify()`** 로 만든다.

**할 일**: `synchronized`; `isAvailable = true`; 출력; **`notify()`**.

<details>
<summary>💡 힌트 보기</summary>

```java
public synchronized void makeResourceAvailable() {
    isAvailable = true;
    System.out.println("Resource is now available!");
    notify();        // ⚠ 일단 notify()로! (기아 현상을 관찰하기 위해)
}
```

지금은 일부러 `notify()`를 씁니다. 다음에 이걸 `notifyAll()`로 바꿔 차이를 볼 거예요.

</details>

**확인**: 공급 시 한 스레드를 깨우는 구조면 OK.

---

### Step 4. 일꾼 스레드 (`WorkerThread`)

**목표**: 일꾼이 끝없이 자원을 요청하고, 얻으면 잠시 작업한다.

<details>
<summary>💡 힌트 보기</summary>

```java
class WorkerThread extends Thread {
    private SharedResource resource;
    private String name;
    public WorkerThread(SharedResource resource, String name) {
        this.resource = resource;
        this.name = name;
    }
    public void run() {
        while (true) {
            resource.waitForResource(name);
            try { Thread.sleep(1000); }     // 자원 얻은 뒤 1초 작업
            catch (InterruptedException e) { break; }
        }
    }
}
```

</details>

**확인**: 같은 `resource`를 받아 계속 요청하는 구조면 OK.

---

### Step 5. main 실행 — 기아 현상 관찰 (notify 버전)

**목표**: 일꾼 **3명** + **2초마다 자원을 공급**하는 스레드를 실행하고, `notify()`일 때 기아를 관찰한다.

**할 일**
1. `SharedResource` 하나를 만들어 일꾼 3명에게 공유.
2. 2초마다 `makeResourceAvailable()`을 호출하는 공급자 스레드.

<details>
<summary>💡 힌트 보기</summary>

```java
public static void main(String[] args) {
    SharedResource resource = new SharedResource();   // 하나의 공유 자원

    new WorkerThread(resource, "worker1").start();
    new WorkerThread(resource, "worker2").start();
    new WorkerThread(resource, "worker3").start();

    // 2초마다 자원 공급
    new Thread(() -> {
        while (true) {
            try {
                Thread.sleep(2000);
                resource.makeResourceAvailable();
            } catch (InterruptedException e) {
                throw new RuntimeException(e);
            }
        }
    }).start();
}
```

</details>

**관찰 포인트**: 출력에서 `got the resource!`를 얻는 일꾼을 살펴보세요. `notify()`라서 **특정 일꾼이 거의 못 얻고 계속 "waiting..."만** 하는 경향(기아)이 보일 수 있어요. (실행마다 다름)

---

### Step 6. notifyAll()로 해결

**목표**: `notify()` → `notifyAll()`로 바꿔, 모든 일꾼이 고르게 기회를 얻게 한다.

**할 일**: `makeResourceAvailable`의 `notify()`를 `notifyAll()`로 교체.

<details>
<summary>💡 힌트 보기</summary>

```java
public synchronized void makeResourceAvailable() {
    isAvailable = true;
    System.out.println("Resource is now available!");
    notifyAll();     // ✅ 모두 깨워 공정하게 경쟁
}
```

</details>

**관찰 포인트**: 이제 worker1·2·3이 **돌아가며 고르게** `got the resource!`를 얻는지 비교. notify() 때와 달리 굶는 일꾼이 줄어듭니다.

## 8. (선택) 도전 과제

1. **공급량 늘리기**: 한 번에 자원을 여러 개(`int count`) 공급하고, 그만큼 깨우면 어떻게 달라지는지
2. **공정한 큐**: 일꾼들을 **줄 세워서**(요청 순서대로) 자원을 주도록 직접 순번 관리 → notifyAll보다 더 공정하게
3. **통계 내기**: 각 일꾼이 자원을 몇 번 얻었는지 카운트해서, notify() vs notifyAll()의 분배 차이를 **숫자로** 비교
4. **fair lock 비교**: `ReentrantLock`의 `Condition` + 공정 모드로 같은 문제를 풀어, notifyAll보다 공정성이 좋은지 관찰
5. **기아를 일부러 심하게**: 일꾼 수를 10명으로 늘려 notify()의 기아가 더 두드러지는지 실험
