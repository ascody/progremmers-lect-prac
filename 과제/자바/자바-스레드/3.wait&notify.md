# wait() & notify() 익히기 (스레드 협업)

> `synchronized`로 자원을 보호하는 걸 배웠다면, 이번엔 스레드가 **서로 차례를 주고받는** 방법을 익힙니다.
> `wait()`은 "락을 내려놓고 잠시 비켜주기", `notify()`는 "이제 네 차례야 하고 깨우기"예요.
> 질문과 답변이 **번갈아** 출력되는 채팅을 만들며 체득합니다.
> **아래 Step을 순서대로 따라가면, 두 스레드가 사이좋게 차례를 주고받게 됩니다.**
>
> 💡 각 Step의 **힌트는 접혀 있어요.** 먼저 스스로 해보고, 막히면 "힌트 보기"를 펼치세요.

---

## 0. 먼저 알아둘 점

`wait()`/`notify()`는 **반드시 `synchronized` 안에서만** 호출할 수 있습니다(그 객체의 락을 가진 상태여야 하므로).
또한 스레드 결과는 비결정적일 수 있지만, 이 예제는 flag로 차례를 강제하므로 **질문 → 답변 → 질문 → 답변** 순서로 깔끔하게 번갈아 나오는 게 목표예요.

---

## 1. 무엇을 하나요?

질문 스레드와 답변 스레드가 하나의 `Chat`을 공유하며, 서로 차례를 주고받아 번갈아 출력합니다.

```
Question : Hi
Answer : Hello
Question : How are you?
Answer : I'm fine, thank you!
Question : What are you doing?
Answer : I'm coding in Java
```

---

## 2. 학습 목표

| 개념 | 어디서 배우나 |
|------|------|
| `wait()` | 락을 내려놓고 대기실에서 기다리기 |
| `notify()` | 대기 중인 스레드 깨우기 |
| `synchronized` 필수 | wait/notify는 동기화 블록 안에서만 |
| `while`로 조건 재확인 | 깨어난 뒤 조건을 다시 검사 |
| flag로 차례 관리 | 질문 차례 / 답변 차례 |

---

## 3. 핵심 개념

### (1) 왜 wait/notify가 필요할까?
`synchronized`만으로는, 조건이 안 맞아 더 진행 못 하는 스레드가 **락을 쥔 채 버티면** 다른 스레드가 전부 막혀요. (예: 출금할 돈이 없는데 락을 잡고 입금만 기다리는 경우)
→ 이때 `wait()`으로 **락을 내려놓고 비켜주면**, 다른 스레드가 일을 하고, 조건이 갖춰지면 `notify()`로 깨워줍니다.

### (2) ⭐ wait() vs sleep() — 결정적 차이
| | `sleep()` | `wait()` |
|---|---|---|
| 멈추는가 | O | O |
| **락(lock)은?** | **쥔 채로 잔다** | **내려놓고 잔다** |
| 깨우는 법 | 시간이 지나면 | `notify()` / `notifyAll()` |

> 핵심: `wait()`은 **락을 반납**하기 때문에, 자는 동안 다른 스레드가 그 객체에 들어가 일할 수 있어요. `sleep()`은 락을 쥐고 자서 아무도 못 들어옵니다.

### (3) 반드시 synchronized 안에서
`wait()`/`notify()`는 **그 객체의 락을 가진 상태**에서만 호출할 수 있어요. 그래서 항상 `synchronized` 메서드나 블록 안에서 씁니다. (안 그러면 `IllegalMonitorStateException`)

### (4) 왜 `if`가 아니라 `while`?
깨어났다고 해서 **조건이 반드시 맞는 건 아니에요.** 가짜로 깨어나거나, 여러 스레드가 경쟁할 수 있거든요. 그래서 깨어난 뒤 **조건을 다시 검사**해야 하고, 그러려면 `if`가 아니라 `while`로 감싸야 합니다.
```java
while (조건) { wait(); }   // 깨어나도 다시 조건 확인 → 안전
```

### (5) notify vs notifyAll
- `notify()` — 대기실의 스레드 중 **임의로 하나만** 깨움
- `notifyAll()` — **전부** 깨우지만, 락은 결국 하나만 얻고 나머지는 다시 대기

### Step 1. 공유 객체 Chat 골격

**목표**: 질문/답변이 함께 쓰는 `Chat`과, 차례를 나타내는 `flag`를 만든다.

**할 일**: `boolean flag` 필드. (`false` = 질문 차례, `true` = 답변 차례)

<details>
<summary>💡 힌트 보기</summary>

```java
class Chat {
    private boolean flag = false;   // false: 질문 차례, true: 답변 차례
}
```

</details>

**확인**: 컴파일 에러가 없으면 OK.

---

### Step 2. 질문 메서드 — question()

**목표**: 질문 차례가 아니면 기다리고, 차례가 오면 질문을 출력한 뒤 답변에게 차례를 넘긴다.

**할 일**
1. `synchronized` 메서드.
2. `flag`가 `true`(답변 차례)면 `wait()`.
3. 질문 출력 → `flag = true` → `notify()`.

<details>
<summary>💡 힌트 보기</summary>

```java
public synchronized void question(String msg) {
    while (flag) {                  // 아직 답변 차례면
        try {
            wait();                 // 락 내려놓고 대기
        } catch (InterruptedException e) {
            throw new RuntimeException(e);
        }
    }
    System.out.println("Question : " + msg);
    flag = true;                    // 이제 답변 차례
    notify();                       // 답변 스레드 깨우기
}
```

`while (flag)` 로 감싸는 걸 잊지 마세요! 깨어난 뒤 조건을 다시 확인해야 합니다.

</details>

**확인**: (실행은 Step 5에서) 질문 출력 후 flag가 넘어가는 구조가 되면 OK.

---

### Step 3. 답변 메서드 — answer()

**목표**: question과 **대칭**으로, 답변 차례가 아니면 기다리고, 차례가 오면 답변을 출력한 뒤 질문에게 차례를 넘긴다.

**할 일**: `flag`가 `false`(질문 차례)면 `wait()`; 답변 출력 → `flag = false` → `notify()`.

<details>
<summary>💡 힌트 보기</summary>

```java
public synchronized void answer(String msg) {
    while (!flag) {                 // 아직 질문 차례면
        try {
            wait();
        } catch (InterruptedException e) {
            throw new RuntimeException(e);
        }
    }
    System.out.println("Answer : " + msg);
    flag = false;                   // 다시 질문 차례
    notify();                       // 질문 스레드 깨우기
}
```

question과 비교해 보세요. 조건이 `flag` → `!flag`, 마지막 flag가 `true` → `false`로 **거울처럼 대칭**이에요.

</details>

**확인**: question과 answer가 서로 반대 차례를 기다리는 구조가 되면 OK.

---

### Step 4. 질문/답변 스레드

**목표**: 각자 메시지 배열을 돌며 `chat.question(...)` / `chat.answer(...)`를 호출하는 스레드를 만든다.

<details>
<summary>💡 힌트 보기</summary>

```java
class QuestionThread extends Thread {
    private Chat chat;
    private String[] questions = { "Hi", "How are you?", "What are you doing?" };
    public QuestionThread(Chat chat) { this.chat = chat; }
    public void run() {
        for (String q : questions) {
            chat.question(q);
            try { Thread.sleep(500); } catch (InterruptedException e) { break; }
        }
    }
}

class AnswerThread extends Thread {
    private Chat chat;
    private String[] answers = { "Hello", "I'm fine, thank you!", "I'm coding in Java" };
    public AnswerThread(Chat chat) { this.chat = chat; }
    public void run() {
        for (String a : answers) {
            chat.answer(a);
            try { Thread.sleep(500); } catch (InterruptedException e) { break; }
        }
    }
}
```

</details>

**확인**: 두 스레드 모두 같은 `chat`을 받아 쓰도록 생성자가 되어 있으면 OK.

---

### Step 5. main 실행 — 하나의 Chat을 공유

**목표**: **하나의 Chat 객체**를 두 스레드가 함께 쓰게 하고 실행한다.

<details>
<summary>💡 힌트 보기</summary>

```java
public static void main(String[] args) {
    Chat chat = new Chat();              // 단 하나의 공유 객체!

    QuestionThread qt = new QuestionThread(chat);
    AnswerThread at = new AnswerThread(chat);

    qt.start();
    at.start();
}
```

**중요**: Chat을 **하나만** 만들어 둘에게 같이 넘겨야 해요. 각자 다른 Chat을 만들면 차례를 주고받을 수 없습니다.

</details>

**확인**: `Question : ...` 와 `Answer : ...` 가 **번갈아** 출력되면 성공! 🎉

## 8. (선택) 도전 과제

1. **`while`을 `if`로 바꿔보기**: 조건 검사를 `if`로 바꾸면 어떤 문제가 생기는지 관찰 (특히 스레드를 더 늘렸을 때) → 왜 `while`이 정답인지 체감
2. **`wait()` vs `sleep()` 실험**: question에서 `wait()` 대신 `Thread.sleep()`을 써보고, 다른 스레드가 못 들어와 멈추는 걸 관찰 (락을 안 놓기 때문)
3. **notifyAll로 바꾸기**: 질문 스레드 2개 + 답변 스레드 2개로 늘리고 `notify()` → `notifyAll()`로 바꿔 안정적으로 동작시키기
4. **생산자-소비자**: 채팅 대신 "빵을 만드는 스레드 / 먹는 스레드"로 바꿔, 빵이 없으면 먹는 쪽이 wait, 만들면 notify 하는 구조 구현
5. **시간 제한 wait**: `wait(2000)`처럼 시간 지정 wait을 써서 "최대 2초만 기다리기" 실험
