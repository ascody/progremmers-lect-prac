# 전략 패턴 리팩토링 (별도 클래스 → 익명 클래스 → 람다)

> 전략 패턴에서 **변하는 부분(전략)** 을 어떻게 끼워 넣을지를, **별도 클래스 → 익명 클래스 → 람다** 순으로 줄여가며 익혀요.
> 핵심은 "왜 익명 클래스가 편한가"예요. 답은 **지역 변수 캡처** — 직접 보면 바로 와닿아요.
> **아래 Step을 순서대로 따라가면, 마지막 Step에서 같은 동작을 세 가지 방식으로 쓸 수 있게 됩니다.**
>
> 💡 각 Step의 **힌트는 접혀 있어요.** "이 별도 클래스를 어떻게 메서드 안으로 넣지?"를 먼저 고민하고, 막히면 펼치세요.

---

## 0. 먼저 알아둘 점

- 전략 패턴은 코드를 **둘로 나눠요**: **컨텍스트(context)** = 변하지 않는 공통 흐름(자원 열기·실행·정리), **전략(strategy)** = 변하는 부분(무슨 작업을 할지). 컨텍스트는 **인터페이스에만 의존**하고, 실제 전략은 **런타임에 주입**받아요. 그래서 새 기능을 추가해도 컨텍스트는 안 고치고(닫힘) 전략만 새로 만들면 돼요(열림) = **OCP**.
- 원래 예제는 JDBC(Connection/PreparedStatement)였지만, 실제 DB 없이 돌릴 수 있게 **인메모리로 단순화**했어요. 패턴 구조는 똑같아요. 대응은 이래요.

  | 원래(JDBC) | 이 과제(인메모리) |
  |---|---|
  | `jdbcContextWithStatementStrategy` | `context` |
  | `StatementStrategy.makeStatement(conn)` + `executeUpdate` | `StatementStrategy.run(db)` (전략이 작업을 직접 수행) |
  | `Connection`/자원정리 | `Database.open()` / `close()` |
  | 별도 전략 클래스 `UserDAODeleteAll` 등 | `DeleteAllStrategy` 등 |

- 인터페이스, 익명 클래스, 람다는 안다고 가정해요. (`StatementStrategy`는 추상 메서드가 1개라 **함수형 인터페이스** → 람다로 쓸 수 있어요.)

---

## 1. 무엇을 만드나요?

같은 컨텍스트(열기→전략→닫기)에 **전략만 바꿔 끼우는** 프로그램을 완성해요. 전략을 세 가지 방식으로 만들어봐요.

```
== (별도 클래스) deleteAll ==
  [컨텍스트] 연결 열기
  [전략-별도클래스] 전체 삭제
  [컨텍스트] 연결 닫기

== (익명 클래스) add(김) ==
  [컨텍스트] 연결 열기
  [전략-익명] 추가: 김
  [컨텍스트] 연결 닫기

== (람다) add(이) ==
  [컨텍스트] 연결 열기
  [전략-람다] 추가: 이
  [컨텍스트] 연결 닫기

현재 사용자 수: 2
사용자: 김
사용자: 이
```

세 경우 모두 **"연결 열기 → (전략) → 연결 닫기"** 흐름은 똑같죠? **변하지 않는 부분은 컨텍스트가, 변하는 부분만 전략이** 담당하는 게 핵심이에요.

## 2. 학습 목표

| 개념 | 어디서 배우나 |
|------|------|
| 전략 인터페이스 + 컨텍스트(공통 흐름) | Step 1 (`StatementStrategy.java`, `UserDao.java`) |
| 전략을 별도 클래스로 분리 | Step 2 (`DeleteAllStrategy.java`, `AddStrategy.java`) |
| 익명 클래스로 인라인 + 지역변수 캡처 | Step 3 (`UserDao.java`) |
| 람다로 더 줄이기 | Step 4 (`UserDao.java`) |
| 전체 실행 | Step 5 (`Main.java`) |

## 3. 핵심 개념

### (1) 전략 패턴 = 컨텍스트(불변) + 전략(가변)
```
컨텍스트(context)            전략(strategy)
─────────────────           ──────────────
db.open();        ┐         "무슨 작업?"  ← 여기만 바뀐다
strategy.run(db); ┼ 공통     · 전체 삭제
db.close();       ┘         · 사용자 추가
```
컨텍스트는 `StatementStrategy`라는 **인터페이스에만** 의존해요. 그래서 새 작업이 생겨도 컨텍스트는 그대로 두고 **전략만 새로** 만들면 돼요.

### (2) 별도 클래스의 불편함
전략을 별도 클래스로 만들면, 작업마다 **클래스 파일이 하나씩** 생겨요. 게다가 작업에 **데이터가 필요하면**(예: add할 `user`) 클래스는 그 값을 **생성자로 받아 필드에 저장**해야 해요. 한 번 쓰고 마는 작업치고는 번거롭죠.

### (3) 익명 클래스의 장점 = 지역 변수 캡처
익명 클래스는 메서드 **안에 바로** 쓰니 별도 파일이 안 생겨요. 더 중요한 건, 바깥 메서드의 **지역 변수를 그대로 가져다 쓸 수 있다(캡처)** 는 거예요. 그래서 `user`를 생성자로 넘길 필요 없이 바로 써요.
```java
// 별도 클래스: user를 생성자로 받아야 함
new AddStrategy(user);

// 익명 클래스: user를 그냥 캡처해서 씀 (생성자 불필요!)
new StatementStrategy() {
    public void run(Database db) { db.getUsers().add(user); }  // 바깥 user를 그대로
};
```

### (4) 람다 = 함수형 인터페이스면 더 짧게
`StatementStrategy`는 추상 메서드가 1개라, 익명 클래스를 **람다로** 줄일 수 있어요.
```java
context(db -> db.getUsers().add(user));   // 익명 클래스의 껍데기를 다 벗김
```

```
[기억법] 변하는 건 전략, 안 변하는 건 컨텍스트 · 별도클래스 → 익명(캡처) → 람다 로 줄이기
```

## 4. 파일 구조

| 파일 | 역할 |
|------|------|
| `User.java` | 사용자 도메인 — id/name |
| `Database.java` | (제공) 인메모리 저장 + 열기/닫기 흉내 |
| `StatementStrategy.java` | 전략 인터페이스 — `void run(Database db)` |
| `DeleteAllStrategy.java` / `AddStrategy.java` | (Step 2) 별도 클래스 전략 |
| `UserDao.java` | 컨텍스트 + deleteAll/add |
| `Main.java` | 시나리오 실행 |

> 실행: `javac *.java` → `java -Dstdout.encoding=UTF-8 Main`

## 5. Step by Step

### Step 1. 전략 인터페이스 + 컨텍스트 만들기 (`StatementStrategy.java`, `UserDao.java`)

**목표**: 변하지 않는 공통 흐름(컨텍스트)과, 변하는 부분의 틀(전략 인터페이스)을 만들어요.

`Database`(제공)는 진짜 DB 대신 리스트에 저장하고, 열기/닫기를 흉내 내요.
```java
// Database.java (제공)
import java.util.ArrayList;
import java.util.List;
class Database {
    private List<User> users = new ArrayList<>();
    void open()  { System.out.println("  [컨텍스트] 연결 열기"); }
    void close() { System.out.println("  [컨텍스트] 연결 닫기"); }
    List<User> getUsers() { return users; }
}
```

**할 일**
1. `StatementStrategy` 인터페이스를 만들어요. 메서드는 `void run(Database db);`
2. `UserDao`에 `Database`를 주입받고, **컨텍스트** `context(StatementStrategy strategy)`를 만들어요. (열기 → `strategy.run(db)` → 닫기)

<details>
<summary>💡 힌트 보기</summary>

```java
// StatementStrategy.java
@FunctionalInterface
interface StatementStrategy {
    void run(Database db);          // 변하는 부분
}
```
```java
// UserDao.java
class UserDao {
    private Database db;
    UserDao(Database db) { this.db = db; }

    void context(StatementStrategy strategy) {
        db.open();                  // 공통: 자원 열기
        strategy.run(db);           // 변하는 부분을 전략에 위임
        db.close();                 // 공통: 자원 정리
    }
}
```

`context`는 **전략이 무엇이든 신경 안 써요.** `StatementStrategy` 인터페이스만 보고 `run`을 호출할 뿐이에요. 이게 "컨텍스트는 인터페이스에만 의존한다"의 의미예요.

</details>

**확인**: `context`가 컴파일되고, 어떤 전략이든 받아 열기→실행→닫기를 돌릴 준비가 되면 통과예요.

---

### Step 2. 전략을 '별도 클래스'로 만들기 (`DeleteAllStrategy.java`, `AddStrategy.java`)

**목표**: 리팩토링 *전* 모습 — 전략마다 클래스를 따로 만들어요. (불편함을 직접 느껴봐요.)

**할 일**
1. `DeleteAllStrategy`(`StatementStrategy` 구현)를 만들어, `run`에서 전체 삭제를 해요.
2. `AddStrategy`를 만들어, 추가할 `user`를 **생성자로 받아 필드에 저장**하고 `run`에서 추가해요.
3. `UserDao`에서 이 전략들을 `new` 해서 `context`에 넘기는 `deleteAll()`, `add(user)`를 만들어요.

<details>
<summary>💡 힌트 보기</summary>

```java
// DeleteAllStrategy.java
class DeleteAllStrategy implements StatementStrategy {
    public void run(Database db) {
        db.getUsers().clear();
        System.out.println("  [전략-별도클래스] 전체 삭제");
    }
}
// AddStrategy.java
class AddStrategy implements StatementStrategy {
    private final User user;                 // ← 캡처를 못 하니 필드에 보관
    AddStrategy(User user) { this.user = user; }
    public void run(Database db) {
        db.getUsers().add(user);
        System.out.println("  [전략-별도클래스] 추가: " + user.getName());
    }
}
```
```java
// UserDao.java 안에서
void deleteAll() { context(new DeleteAllStrategy()); }
void add(User user) { context(new AddStrategy(user)); }   // user를 생성자로 넘김
```

`AddStrategy`가 `user`를 **생성자로 받아 필드에 저장**하는 점을 눈여겨보세요. 별도 클래스라 바깥 변수를 직접 못 쓰기 때문이에요. 이 번거로움을 Step 3에서 없애요.

</details>

**확인**: `deleteAll()`, `add(user)`가 동작하고, 콘솔에 "전체 삭제"/"추가: …"가 찍히면 성공이에요.

---

### Step 3. 익명 클래스로 리팩토링 (`UserDao.java`)

**목표**: 별도 클래스를 메서드 **안의 익명 클래스**로 옮겨요. `user`는 **캡처**하니 생성자가 사라져요.

**할 일**
1. `deleteAll()`의 전략을 익명 클래스로 메서드 안에 인라인해요.
2. `add(user)`의 전략도 익명 클래스로 바꾸고, **`user`를 캡처해서** 바로 써요. (`AddStrategy` 생성자 호출이 사라짐)

<details>
<summary>💡 힌트 보기</summary>

```java
void deleteAll() {
    StatementStrategy strategy = new StatementStrategy() {
        @Override
        public void run(Database db) {
            db.getUsers().clear();
            System.out.println("  [전략-익명] 전체 삭제");
        }
    };
    context(strategy);
}

void add(User user) {
    StatementStrategy strategy = new StatementStrategy() {
        @Override
        public void run(Database db) {
            db.getUsers().add(user);                 // ← 바깥의 user를 그대로 캡처!
            System.out.println("  [전략-익명] 추가: " + user.getName());
        }
    };
    context(strategy);
}
```

Step 2의 `AddStrategy` 클래스 파일과 `private final User user; 생성자`가 **통째로 사라졌어요.** 익명 클래스가 바깥 `user`를 캡처하기 때문이에요. (캡처되는 지역 변수는 사실상 final이어야 해요 — 한 번 정해지면 안 바꾸기.)

</details>

**확인**: 별도 전략 클래스 없이도 같은 결과가 나오면 성공이에요. `add`에서 `new AddStrategy(...)`가 없어진 걸 확인하세요.

---

### Step 4. 람다로 한 번 더 줄이기 (`UserDao.java`)

**목표**: 익명 클래스를 **람다**로 줄여요. (`StatementStrategy`가 함수형 인터페이스라 가능)

**할 일**
1. Step 3의 익명 클래스를 람다로 바꿔요. `new StatementStrategy() { ... run ... }` → `db -> { ... }`

<details>
<summary>💡 힌트 보기</summary>

```java
void deleteAll() {
    context(db -> {
        db.getUsers().clear();
        System.out.println("  [전략-람다] 전체 삭제");
    });
}

void add(User user) {
    context(db -> {
        db.getUsers().add(user);                     // user는 여전히 캡처
        System.out.println("  [전략-람다] 추가: " + user.getName());
    });
}
```

`new StatementStrategy() { public void run(Database db) {...} }`의 껍데기(클래스 생성·메서드 이름·타입)가 전부 사라지고 `db -> {...}`만 남았어요. **전략 패턴 + 람다**가 만나면 코드가 이렇게까지 짧아져요.

</details>

**확인**: 람다 버전이 익명 클래스 버전과 똑같이 동작하면 성공이에요.

---

### Step 5. 마무리 — Main에서 실행 (완성!)

**목표**: 세 방식이 모두 같은 컨텍스트 흐름을 타는지 확인해요.

**할 일**
1. `Database`, `UserDao`를 만들어요.
2. `deleteAll`, `add(김)`, `add(이)`를 호출하고, 마지막에 사용자 목록을 출력해요.

<details>
<summary>💡 힌트 보기</summary>

```java
public class Main {
    public static void main(String[] args) {
        Database db = new Database();
        UserDao dao = new UserDao(db);

        dao.deleteAll();
        dao.add(new User("u1", "김"));
        dao.add(new User("u2", "이"));

        System.out.println("\n현재 사용자 수: " + db.getUsers().size());
        for (User u : db.getUsers()) System.out.println("사용자: " + u.getName());
    }
}
```

(어느 스타일을 쓰든 "연결 열기 → 전략 → 연결 닫기" 흐름은 동일하게 나와요.)

</details>

**확인**: `javac *.java` → `java -Dstdout.encoding=UTF-8 Main` 후, 1번 섹션처럼 컨텍스트 흐름이 반복되고 사용자 2명이 출력되면 **완성입니다!** 🎉

## 6. 학습 체크

- [ ] 컨텍스트(불변)와 전략(가변)이 각각 무엇을 맡는지 말할 수 있다
- [ ] 컨텍스트가 "인터페이스에만 의존"한다는 게 왜 OCP인지 안다
- [ ] 별도 클래스 전략이 `user`를 생성자로 받아야 했던 이유를 안다
- [ ] 익명 클래스가 지역 변수를 캡처해 생성자가 필요 없어지는 걸 설명할 수 있다
- [ ] 함수형 인터페이스라 람다로 줄일 수 있는 이유를 안다

## 7. 최종 완성 체크리스트

- [ ] `context`가 `StatementStrategy`에만 의존한다
- [ ] 별도 클래스 → 익명 클래스 → 람다, 세 버전을 직접 만들어봤다
- [ ] 익명/람다 버전에서 `new AddStrategy(...)`가 사라졌다
- [ ] 세 방식 모두 "열기 → 전략 → 닫기" 흐름이 같다
- [ ] 사용자 추가/삭제 결과가 올바르게 나온다

## 8. (선택) 도전 과제

1. `get(id)`(아이디로 한 명 찾기) 전략을 추가해보세요. 컨텍스트는 그대로 두고 전략만 새로 만들면 되는지 확인해요(= OCP 체감).
2. `deleteAll`처럼 **캡처할 값이 없는** 전략은 별도 클래스/익명/람다 중 무엇이 가장 깔끔한지 비교해보세요.
3. 람다 버전에서 본문이 한 줄이면 `{ }`도 줄일 수 있어요. `context(db -> db.getUsers().clear())`처럼 바꿔보세요.
4. ★ 원래 JDBC 코드처럼, 전략이 **"무엇을 할지"를 만들어 돌려주고** 컨텍스트가 **실행**하도록 역할을 나눠보세요. (`StatementStrategy`가 작업 객체를 반환 → 컨텍스트가 실행)
5. ★ 컨텍스트를 `UserDao` 밖의 **별도 클래스(JdbcContext)** 로 분리하고, `UserDao`가 그걸 주입받아 쓰도록 바꿔보세요. (책의 다음 단계 — 컨텍스트의 재사용)
