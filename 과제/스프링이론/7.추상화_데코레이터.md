# 추상화 + 데코레이터 패턴 — 알림 발송 시스템 만들기

## 학습 목표
이번 과제로 두 가지를 **직접 손으로** 증명한다.
1. **추상화**: 알림 채널(이메일/문자/카톡)을 갈아끼워도 **클라이언트 코드는 한 줄도 바뀌지 않는다.**
2. **데코레이터**: 기존 발송 클래스를 **전혀 수정하지 않고** 로깅·재시도·시간측정 같은 부가 기능을 덧입힌다.

> 트랜잭션 예제(UserServiceTx)와 똑같은 구조다. 거기서 배운 "같은 인터페이스 구현 → 같은 타입을 멤버로 보유 → 자기 일 하고 위임"을 새 도메인에 그대로 적용해본다.

---

## 시나리오

우리 서비스는 사용자에게 알림을 보낸다. 처음엔 이메일만 보냈는데, 이제 **문자(SMS)**, **카카오톡**으로도 보내야 한다.
게다가 운영팀이 요구한다:
- "발송할 때마다 **로그**를 남겨줘."
- "네트워크가 불안정하니 실패하면 **자동으로 3번까지 재시도**해줘."
- "느린 채널이 있는지 **발송 시간(ms)**을 재줘."

이 요구를 **각 채널 클래스에 직접 코드를 우겨넣지 않고** 해결하는 게 핵심이다.

## 제공되는 시작 코드

```java
// NotificationSender.java  ── 공통 인터페이스 (수정 금지)
public interface NotificationSender {
    void send(String to, String message);
}
```

```java
// EmailNotificationSender.java  ── 실제 구현체 예시 (그대로 사용)
public class EmailNotificationSender implements NotificationSender {
    @Override
    public void send(String to, String message) {
        // 실제론 메일 서버 호출. 과제에서는 콘솔 출력으로 흉내.
        System.out.printf("[EMAIL] to=%s : %s%n", to, message);
    }
}
```

```java
// FlakyEmailSender.java  ── 재시도 테스트용: 처음 두 번은 일부러 실패한다 (그대로 사용)
public class FlakyEmailSender implements NotificationSender {
    private int attempt = 0;
    @Override
    public void send(String to, String message) {
        attempt++;
        if (attempt < 3) {
            throw new RuntimeException("일시적 네트워크 오류 (시도 " + attempt + ")");
        }
        System.out.printf("[EMAIL] (시도 %d 성공) to=%s : %s%n", attempt, to, message);
    }
}
```

## Part A — 추상화: 채널 갈아끼우기

**A-1.** `SmsNotificationSender`, `KakaoNotificationSender`를 만들어 `NotificationSender`를 구현하라.
(내용은 `EmailNotificationSender`처럼 콘솔에 `[SMS] ...`, `[KAKAO] ...` 형태로 출력하면 된다.)

**A-2.** 알림을 사용하는 클라이언트 `NotificationService`를 작성하라.
이 클래스는 **구체 클래스(Email/Sms/Kakao)를 절대 알면 안 된다.** 오직 `NotificationSender` 인터페이스에만 의존한다.

```java
public class NotificationService {
    private final NotificationSender sender;   // 인터페이스 타입!

    public NotificationService(NotificationSender sender) {
        this.sender = sender;
    }

    public void notifyUser(String to, String message) {
        // TODO: sender 를 이용해 알림을 보낸다
    }
}
```

**A-3. 증명하기.** `main`에서 `NotificationService`에 주입하는 구현체를 Email → Sms → Kakao 로 바꿔가며 실행하라.
**`NotificationService` 코드는 한 글자도 고치지 않은 채** 동작 채널만 바뀌어야 한다.

> ✅ 체크포인트: "구현체를 바꿨는데 클라이언트가 안 바뀐다" — 이게 추상화의 효과다. 왜 가능한지 한 문장으로 설명을 주석에 적어라.

## Part B — 데코레이터: 기존 클래스를 건드리지 않고 기능 추가

**규칙: Part A에서 만든 `EmailNotificationSender` 등 실제 구현체는 절대 수정하지 않는다.** 부가 기능은 전부 데코레이터로 추가한다.

각 데코레이터는 **데코레이터 3요소**를 만족해야 한다.
1. `NotificationSender`를 구현한다
2. `NotificationSender`를 멤버(`delegate`)로 가진다
3. 자기 일을 한 뒤 `delegate.send(...)`로 **위임**한다

**B-1. 로깅 데코레이터** — 발송 전/후에 로그를 남긴다.

```java
public class LoggingNotificationSender implements NotificationSender {
    private final NotificationSender delegate;

    public LoggingNotificationSender(NotificationSender delegate) {
        this.delegate = delegate;
    }

    @Override
    public void send(String to, String message) {
        // TODO 1: "발송 시작" 로그 (to 포함)
        // TODO 2: delegate.send(to, message);   ← 실제 발송은 위임
        // TODO 3: "발송 완료" 로그
    }
}
```

**B-2. 재시도 데코레이터** — 발송이 예외를 던지면 최대 3회까지 재시도하고, 그래도 실패하면 예외를 다시 던진다.
(테스트는 `FlakyEmailSender`를 감싸서 한다.)

**B-3. 시간측정 데코레이터** — 발송에 걸린 시간(ms)을 측정해서 출력한다.
(`System.currentTimeMillis()` 또는 `System.nanoTime()` 사용.)

## Part C — 데코레이터 쌓기 (조합)

데코레이터는 **여러 겹으로 쌓을 수 있다.** 아래처럼 조립해서 실행하라.

```java
NotificationSender sender =
    new TimingNotificationSender(       // ③ 가장 바깥: 전체 소요 시간 측정
        new LoggingNotificationSender(  // ② 로그 남기고
            new RetryNotificationSender(// ① 실패하면 재시도하며
                new FlakyEmailSender())));// (실제 발송 대상)

new NotificationService(sender).notifyUser("user@test.com", "안녕하세요");
```

**C-1.** 위 조합을 실행해 로그·재시도·시간측정이 모두 동작하는지 확인하라.

**C-2. 순서 실험.** `Logging`과 `Retry`의 **감싸는 순서를 서로 바꿔서** 실행해보고, 출력이 어떻게 달라지는지 관찰하라.
- `Logging(Retry(...))` vs `Retry(Logging(...))`
- 재시도할 때 로그가 1번 찍히는가, 3번 찍히는가? 왜 그런가?

> ✅ 체크포인트: 데코레이터의 **순서가 의미를 바꾼다**는 걸 직접 보는 게 이 파트의 핵심이다.

## 도전 과제 (선택)

- **D-1. 속도 제한 데코레이터**: 1초에 1건만 통과시키고, 초과 호출은 잠시 대기시키는 `RateLimitNotificationSender`를 추가하라.
- **D-2. 조건부 발송**: 메시지에 특정 금칙어가 있으면 위임하지 않고 차단하는 데코레이터를 추가하라. (데코레이터가 **위임을 안 할 수도** 있다는 걸 보여줌)
- **D-3. 스프링 DI 조립**: 위 조합을 `@Configuration` + `@Bean`으로 옮겨, 데코레이터 체인을 **설정에서** 구성해보라.
