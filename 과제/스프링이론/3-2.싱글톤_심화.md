# 싱글톤 2편 — 싱글톤 레지스트리와 무상태 (서버 환경의 싱글톤)

> 1편에서 "객체를 하나만 두는 법"을 배웠다면, 이번엔 **"서버에선 왜 싱글톤이 필수이고, 그래서 왜 무상태(stateless)여야 하는가"** 를 익혀요.
> 같은 싱글톤을 **여러 스레드가 동시에** 쓸 때, 요청 데이터를 *필드*에 두면 데이터가 섞이는 걸 **직접 재현**하고 고쳐요.
> **아래 Step을 순서대로 따라가면, 마지막 Step에서 "안전한 서비스 오브젝트"가 완성됩니다.**
>
> 💡 각 Step의 **힌트는 접혀 있어요.** "여러 명이 동시에 쓰면 뭐가 위험하지?"를 먼저 떠올리고, 막히면 펼치세요.

---

## 0. 먼저 알아둘 점

- 이 가이드는 **1편(싱글톤 패턴)** 의 후속이에요. `private` 생성자 / `static` 인스턴스 / `getInstance()` 3종 세트는 안다고 가정해요.
- 이번 주제의 무대는 **웹 서버**예요. 보통 "요청 1개 = 스레드(일꾼) 1개"로 동시에 수백~수천 명을 처리해요. 그래서 **멀티스레드**를 조금 써요. (달팽이 경주에서 배운 `Thread` 수준이면 충분해요.)
- 핵심 메시지는 두 줄이에요.
  1. 요청마다 객체를 `new` 하면 서버가 못 버틴다 → **싱글톤으로 공유**한다.
  2. 하나를 여럿이 공유하니, **요청별 데이터를 필드에 두면 섞인다** → **무상태**로 만든다.
- 스프링을 직접 쓰진 않아요. 대신 스프링이 내부에서 하는 일을 **손으로 작게** 흉내 내며 원리를 잡아요.

## 1. 무엇을 만드나요?

같은 싱글톤을 여러 스레드가 동시에 쓸 때 무슨 일이 나는지 보여주는 프로그램을 완성해요.

```
===== 같은 싱글톤을 30개 스레드가 동시에 사용 =====
[필드에 저장] 데이터 엉킴: 29건 / 30건     <- 위험! 거의 다 섞임
[파라미터로]  데이터 엉킴: 0건 / 30건      <- 안전! 하나도 안 섞임

===== 필드에 둬도 되는 것: 다른 싱글톤 참조 =====
kim 조회 [DB연결]
lee 조회 [DB연결]
같은 DAO인가? true
```

윗부분이 이 가이드의 하이라이트예요. **똑같은 코드 구조인데, 요청 데이터를 "필드"에 두느냐 "파라미터"로 두느냐만으로 결과가 29:0 으로 갈려요.**

## 3. 핵심 개념

### (1) 서버에서 요청마다 `new` 하면 벌어지는 일
요청 한 번을 처리하는 데 객체가 5개 필요하다고 해볼게요.

```
요청 1번        → 5개 생성
초당 500요청    → 초당 2,500개
1분            → 15만 개
1시간          → 9백만 개
```

아무리 자바의 객체 생성과 GC가 빨라도, 이만큼 쏟아지면 서버가 버티기 힘들어요. 그래서 엔터프라이즈에선 일찍부터 **"기능만 제공하는 도구는 하나만 만들어 공유하자"** 는 **서비스 오브젝트** 개념을 써왔어요.

### (2) 서비스 오브젝트와 서블릿, 그리고 ATM 비유
자바 웹의 기본인 **서블릿**이 바로 그래요. 서블릿은 **클래스당 오브젝트 하나**만 만들어두고, 요청을 처리하는 **여러 스레드가 그 하나를 동시에** 써요. (= 멀티스레드 환경의 싱글톤)

> 🏧 **비유**: ATM 한 대(= 서비스 오브젝트)를 여러 손님(= 스레드)이 같이 써요. ATM은 출금·입금 *기능*만 제공할 뿐, 특정 손님의 돈을 자기 안에 보관하지 않아요. 그래서 한 대로 모두가 쓸 수 있어요.

서블릿·DAO 같은 서비스 오브젝트는 **사용자별로 따로 보관할 데이터가 없어요.** 로직만 제공하죠. 그러니 사람마다 새로 만들 이유가 없고, 하나로 공유하면 돼요.

### (3) ⚠️ 그래서 핵심 규칙: 무상태(stateless)
하나를 여럿이 공유한다는 건 무서운 면도 있어요. **요청별 데이터를 인스턴스 변수(필드)에 저장하면**, 거의 동시에 도는 다른 스레드가 그 값을 덮어써서 **데이터가 엉켜요.**

```
공유 필드 name 이 있을 때 (싱글톤 1개를 두 스레드가 동시에):

스레드A: name = "철수"  ──┐
스레드B: name = "영희"     │  ← B가 A의 값을 덮어씀
스레드A: return name   ──┘ → "영희" 가 나옴! (A는 철수였는데)
```

해결은 간단해요. **요청별 데이터는 필드가 아니라 지역변수/파라미터로** 다뤄요. 지역변수는 스레드마다 **각자의 스택**에 따로 생겨서 서로 섞이지 않아요.

### (4) 필드에 둬도 되는 것 vs 안 되는 것

| 구분 | 예 | 필드 OK? |
|------|----|---------|
| 읽기 전용 값 | 변하지 않는 상수, 설정 | ✅ 가능 |
| 다른 싱글톤 참조 | `UserDao`가 가진 `ConnectionMaker` | ✅ 가능 (요청마다 안 바뀜) |
| **요청별 사용자 정보** | 이번 요청의 userId, 이름 | ❌ 위험 → 파라미터로 |
| **처리 중간 계산값** | 합계 누적 변수 등 | ❌ 위험 → 지역변수로 |

### (5) 스프링이 해주는 일: 싱글톤 레지스트리
스프링의 **애플리케이션 컨텍스트(ApplicationContext)** 는 사실 **싱글톤을 저장·관리하는 "싱글톤 레지스트리"** 예요. 별다른 설정이 없으면 스프링은 내부에서 만드는 빈(bean) 오브젝트를 **전부 싱글톤**으로 둬요. 즉, 우리가 이 가이드에서 손으로 만든 `getInstance()` 관리를 **컨테이너가 대신** 해주는 거예요. (지난 IoC 가이드의 "컨테이너"가 바로 이 역할이에요.)

```
[기억법] 서버 싱글톤 = "하나를 모두가 공유(서비스 오브젝트) + 요청 데이터는 필드 말고 파라미터(무상태)"
```

## 4. 파일 구조

| 파일 | 역할 |
|------|------|
| `GreetingServiceBad.java` | (나쁜 예) 요청 데이터를 필드에 저장 → 스레드끼리 엉킴 |
| `GreetingServiceGood.java` | (좋은 예) 파라미터/지역변수만 사용 → 무상태 |
| `Dao.java` | 필드에 둬도 되는 경우: `UserDao`가 다른 싱글톤(`ConnectionMaker`) 참조 |

> 실행: `javac *.java` → `java -Dstdout.encoding=UTF-8 Main`

## 5. Step by Step

### Step 1. 무상태 위반 — 요청 데이터를 필드에 저장하고 엉킴 재현 (`GreetingServiceBad.java`)

**목표**: 싱글톤 서비스가 요청 데이터를 **필드**에 두면 멀티스레드에서 깨진다는 걸 **직접 봐요.**

**할 일**
1. `GreetingServiceBad`를 싱글톤(3종 세트)으로 만들어요.
2. `private String name;` **필드**를 두고, `greet(reqName)`에서 `this.name = reqName` 후 잠깐 멈췄다가 `this.name`을 돌려줘요.
3. (Main에서) 여러 스레드가 각자 다른 이름으로 동시에 호출하게 해서, 돌려받은 이름이 **내가 보낸 이름과 다른** 경우를 세어봐요.

<details>
<summary>💡 힌트 보기</summary>

```java
class GreetingServiceBad {
    private static final GreetingServiceBad instance = new GreetingServiceBad();
    private GreetingServiceBad() {}
    static GreetingServiceBad getInstance() { return instance; }

    private String name;                       // ⚠️ 모든 스레드가 공유하는 필드

    String greet(String reqName) {
        this.name = reqName;                   // 내 요청을 필드에 저장
        try { Thread.sleep(5); } catch (InterruptedException e) {} // 끼어들 틈
        return this.name;                      // 그 사이 덮어써졌을 수 있다!
    }
}
```

`sleep(5)`는 "처리하는 척"이에요. 그 짧은 틈에 다른 스레드가 `this.name`을 바꿔치기 해요. 필드는 **객체에 단 하나뿐**이라, 싱글톤에선 모두가 같은 칸을 공유한다는 게 핵심이에요.

</details>

**확인**: Main 실행 시 `[필드에 저장]`의 엉킴 건수가 **0이 아니면**(대부분 많이 나와요) Step 1 성공이에요. "아, 진짜 섞이는구나"를 본 거예요.

---

### Step 2. 무상태로 고치기 — 파라미터/지역변수만 쓰기 (`GreetingServiceGood.java`)

**목표**: 같은 기능을, **필드 없이** 만들어서 멀티스레드에서도 안 섞이게 해요.

**할 일**
1. `GreetingServiceGood`도 싱글톤으로 만들되, **요청 데이터를 담는 필드를 두지 않아요.**
2. `greet(reqName)`에서 받은 **파라미터만** 써서 결과를 만들어요.

<details>
<summary>💡 힌트 보기</summary>

```java
class GreetingServiceGood {
    private static final GreetingServiceGood instance = new GreetingServiceGood();
    private GreetingServiceGood() {}
    static GreetingServiceGood getInstance() { return instance; }

    // 요청 데이터를 담는 필드가 '없다'

    String greet(String reqName) {             // reqName = 파라미터(지역) → 스레드마다 따로
        try { Thread.sleep(5); } catch (InterruptedException e) {}
        return reqName;                        // 절대 안 섞인다
    }
}
```

`reqName`은 메서드 **파라미터**라, 호출한 스레드의 **스택에 따로** 만들어져요. 다른 스레드가 손댈 수 없으니 안전해요. **싱글톤이라도 무상태면 안전하다** — 이게 서비스 오브젝트의 핵심이에요.

</details>

**확인**: Main 실행 시 `[파라미터로]`의 엉킴 건수가 **0건**이면 성공이에요.

---

### Step 3. 필드에 둬도 되는 것 — 다른 싱글톤 참조하기 (`Dao.java`)

**목표**: "필드 = 무조건 나쁨"이 아니라, **요청마다 안 바뀌는 협력자**는 필드여도 된다는 걸 익혀요.

**할 일**
1. `ConnectionMaker` 인터페이스와, 그걸 구현한 **싱글톤** `SimpleConnectionMaker`를 만들어요.
2. `UserDao`(싱글톤)가 `ConnectionMaker`를 **필드로** 가지게 해요. (✅ 안전)
3. 단, `findUser(userId)`처럼 **요청별 데이터(userId)는 파라미터로** 받아요. (❌ 필드 금지)

<details>
<summary>💡 힌트 보기</summary>

```java
interface ConnectionMaker { String makeConnection(); }

class SimpleConnectionMaker implements ConnectionMaker {     // 이것도 싱글톤
    private static final SimpleConnectionMaker instance = new SimpleConnectionMaker();
    private SimpleConnectionMaker() {}
    static SimpleConnectionMaker getInstance() { return instance; }
    public String makeConnection() { return "DB연결"; }
}

class UserDao {
    private static final UserDao instance = new UserDao();
    private UserDao() {}
    static UserDao getInstance() { return instance; }

    private ConnectionMaker connectionMaker = SimpleConnectionMaker.getInstance(); // ✅ 필드 OK

    String findUser(String userId) {            // ✅ 요청별 데이터는 파라미터
        return userId + " 조회 [" + connectionMaker.makeConnection() + "]";
    }
}
```

`connectionMaker`는 **요청마다 바뀌지 않는 읽기 전용 협력자**(다른 싱글톤)예요. 그래서 필드여도 안 섞여요. 반면 `userId`는 사람마다 달라서 **파라미터**로 받아야 해요. 이 구분이 핵심이에요.

</details>

**확인**: `UserDao.getInstance()`가 항상 같은 객체이고(`==` true), `findUser`가 사용자별로 올바른 결과를 주면 성공이에요.

---

### Step 4. 마무리 — Main에서 멀티스레드로 비교 실행 (`Main.java`)

**목표**: 나쁜 서비스와 좋은 서비스를 **같은 조건(여러 스레드 동시)** 에서 돌려 결과를 비교해요.

**할 일**
1. 스레드 `N`개를 만들어, 각자 다른 이름으로 `GreetingServiceBad`를 호출하고 **엉킨 횟수**를 세요.
2. 똑같이 `GreetingServiceGood`로도 돌려 비교해요.
3. `UserDao`로 Step 3도 확인해요.

<details>
<summary>💡 힌트 보기</summary>

```java
public class Main {
    static int badMismatch = 0;
    static int goodMismatch = 0;

    public static void main(String[] args) throws InterruptedException {
        int N = 30;

        // 나쁜 서비스: N개 스레드 동시 실행
        Thread[] t1 = new Thread[N];
        for (int i = 0; i < N; i++) {
            final String myName = "손님" + i;
            t1[i] = new Thread(() -> {                       // 스레드가 할 일
                String r = GreetingServiceBad.getInstance().greet(myName);
                if (!r.equals(myName)) {                     // 내 이름이 안 돌아오면 = 엉킴
                    synchronized (Main.class) { badMismatch++; }
                }
            });
        }
        for (Thread t : t1) t.start();   // 동시에 출발
        for (Thread t : t1) t.join();    // 다 끝날 때까지 대기

        // 좋은 서비스도 똑같은 방식으로(t2, goodMismatch) 반복...

        System.out.println("[필드에 저장] 엉킴: " + badMismatch + " / " + N);
        System.out.println("[파라미터로]  엉킴: " + goodMismatch + " / " + N);
    }
}
```

`start()`로 동시에 출발시키고 `join()`으로 전부 끝나길 기다리는 게 포인트예요. `synchronized`는 카운터(`badMismatch`)를 셀 때 그 값도 안 엉키게 하려는 거예요. (이 하네스 코드 자체는 그냥 "테스트 도구"이고, 진짜 배울 건 서비스 클래스 둘의 차이예요.)

</details>

**확인**: `[필드에 저장]`은 엉킴이 많이 나오고, `[파라미터로]`는 **0건**이면 **완성입니다!** 🎉

## 6. 학습 체크

- [ ] 서버에서 요청마다 `new` 하면 왜 위험한지 숫자로 설명할 수 있다
- [ ] "서비스 오브젝트"가 왜 사람마다 새로 만들 필요가 없는지 안다 (ATM 비유)
- [ ] 싱글톤 필드에 요청 데이터를 두면 왜 섞이는지 설명할 수 있다
- [ ] 지역변수/파라미터가 왜 안 섞이는지(스레드별 스택) 안다
- [ ] 필드에 둬도 되는 것 / 안 되는 것을 구분할 수 있다
