# 싱글톤 패턴 익히기 (프로그램에 객체를 딱 하나만 두기)

> "싱글톤(Singleton)"을, **번호표 발급기**가 두 대 생겼을 때 번호가 겹치는 버그를 직접 재현하고 → 하나로 만들어 고치며 익혀요.
> *왜* 하나만 있어야 하는지를 버그로 먼저 느끼고, 그걸 막는 3종 세트(private 생성자 / static 인스턴스 / getInstance)를 손에 익혀요.
> **아래 Step을 순서대로 따라가면, 마지막 Step에서 싱글톤이 동작하는 프로그램이 완성됩니다.**
>
> 💡 각 Step의 **힌트는 접혀 있어요.** "어떻게 하면 외부에서 객체를 못 만들게 하지?"를 먼저 고민하고, 막히면 펼치세요.

---

## 0. 먼저 알아둘 점

- 싱글톤은 **"이 클래스의 객체는 프로그램 전체에서 단 1개"** 를 보장하는 패턴이에요. 번호표 기계, 설정 관리자, 로그 기록기처럼 **여러 개면 곤란한 것**에 써요.
- 핵심은 딱 **3가지**예요: ① 생성자를 `private`으로 막고, ② `static`으로 유일한 객체를 들고, ③ `getInstance()`라는 유일한 출입구로만 꺼내 쓰기.
- 만드는 시점에 따라 두 가지 방식이 있어요: **eager(이른 초기화)** 와 **lazy(늦은 초기화)**. 둘 다 다뤄요.
- ⚠️ **정직한 한계**: lazy 방식은 **여러 스레드가 동시에** `getInstance()`를 부르면 객체가 2개 만들어질 수 있어요. 이 가이드는 단일 스레드를 가정하고, 그 해결(`synchronized` 등)은 **도전 과제**로 빼요.
- `static`, `private`, 참조 비교 `==`는 안다고 가정해요. (모르면 그 부분만 따로 짚어줄게요.)

---

## 1. 무엇을 만드나요?

"하나만 있어야 정상"인 상황을, 버그 → 해결 순서로 보여주는 `Main`을 완성해요. 다 만들면 이렇게 나와요.

```
===== 1. 싱글톤 없이: 번호표 두 대 (버그!) =====
A 기계가 발급: 1번
B 기계가 발급: 1번  <- 중복!
A 기계가 발급: 2번
B 기계가 발급: 2번  <- 또 중복!

===== 2. 싱글톤 적용: 번호표는 하나뿐 =====
1번 창구가 발급: 1번
2번 창구가 발급: 2번
1번 창구가 발급: 3번
3번 창구가 발급: 4번
같은 기계인가? true

===== 3. lazy 초기화 (설정 관리자) =====
앱 설정 - 테마: dark
앱 설정 - 테마: dark (어디서 불러도 같은 설정)
같은 설정 객체인가? true
```

1번에서 **번호가 겹치는 버그**가 보이죠? 이게 "객체가 여러 개라서" 생기는 문제예요. 2번에서 싱글톤으로 고쳐요.

## 3. 핵심 개념

### (1) 싱글톤이 푸는 문제
번호표 기계가 **두 대** 있으면, 둘 다 "1번"부터 발급해서 **같은 번호의 손님이 둘** 생겨요. 설정 관리자가 두 개면 한쪽 설정만 바꿔도 다른 쪽은 모르고요. **"전체가 공유해야 하는 단 하나의 상태"** 가 있을 때, 객체를 하나로 못 박는 게 싱글톤이에요.

### (2) 싱글톤 3종 세트
```
class TicketMachine {
    ① private static final TicketMachine instance = new TicketMachine();  // 유일한 객체를 static으로 보관
    ② private TicketMachine() {}                                          // 생성자 private → 밖에서 new 금지
    ③ static TicketMachine getInstance() { return instance; }             // 유일한 출입구
}
```
- ② `private` 생성자 덕분에 외부에서 `new TicketMachine()`이 **컴파일 에러**가 나요. (그래서 못 늘려요.)
- ① 객체는 `static`이라 **클래스당 하나**만 존재해요.
- ③ 그 하나를 꺼내는 통로는 `getInstance()` 뿐이에요. 누가 어디서 불러도 **같은 객체**를 돌려받아요.

### (3) eager vs lazy — 언제 만드나
```
[eager] 클래스가 로딩될 때 미리 만들어 둠.    간단하고 안전. (이 가이드의 기본)
   private static final X instance = new X();

[lazy ] getInstance()가 처음 불릴 때 만듦.     안 쓰면 안 만들어 메모리 절약,
   if (instance == null) instance = new X();    단 멀티스레드에서 주의 필요!
```

```
[기억법] 싱글톤 = "new는 나만(private), 꺼내는 건 getInstance로, 객체는 static 하나"
```

## 5. Step by Step

### Step 1. (출발점) 객체가 여러 개일 때의 버그 재현 (`NaiveTicketMachine.java`)

**목표**: 싱글톤이 *없을 때* 무슨 일이 나는지 **직접 겪어봐요.**

**할 일**
1. 평범한 `NaiveTicketMachine`을 만들어요. `issue()`를 부를 때마다 번호를 1씩 올려 돌려줘요.
2. `main`에서 **두 대**(`a`, `b`)를 만들어 번갈아 발급해보고, 번호가 어떻게 되는지 관찰해요.

<details>
<summary>💡 힌트 보기</summary>

```java
// NaiveTicketMachine.java
class NaiveTicketMachine {
    private int lastNumber = 0;
    int issue() { lastNumber++; return lastNumber; }
}
```
```java
// main 안에서
NaiveTicketMachine a = new NaiveTicketMachine();
NaiveTicketMachine b = new NaiveTicketMachine();
System.out.println(a.issue());   // 1
System.out.println(b.issue());   // 1  ← b는 자기만의 lastNumber라 또 1!
```

기계가 두 대니까 `lastNumber`도 **각자** 가져요. 그래서 번호가 겹쳐요. 이게 "객체가 여러 개"라서 생기는 문제예요.

</details>

**확인**: A·B가 둘 다 "1번"을 발급하는 **중복 버그**를 직접 확인하면 통과예요. 이걸 Step 2에서 막아요.

---

### Step 2. 싱글톤 3종 세트 적용하기 (`TicketMachine.java`)

**목표**: 발급기를 **단 하나**만 존재하도록 막아요. (eager 방식)

**할 일**
1. 생성자를 `private`으로 만들어 **외부에서 `new` 못 하게** 해요.
2. `private static final`로 **유일한 객체**를 클래스 안에 하나 보관해요.
3. `static getInstance()`로 그 객체를 꺼내는 **유일한 통로**를 만들어요.

<details>
<summary>💡 힌트 보기</summary>

```java
class TicketMachine {
    private static final TicketMachine instance = new TicketMachine(); // ① 유일 객체
    private int lastNumber = 0;

    private TicketMachine() {}                                 // ② new 차단
    static TicketMachine getInstance() { return instance; }    // ③ 유일 출입구

    int issue() { lastNumber++; return lastNumber; }
}
```

이제 `new TicketMachine()`은 **컴파일 에러**가 나요(생성자가 `private`이라서). 객체를 얻는 방법은 `TicketMachine.getInstance()` 하나뿐이에요.

</details>

**확인**: `main`에서 `new TicketMachine()`을 시도하면 빨간 줄(컴파일 에러)이 뜨고, `getInstance()`로만 얻어지면 성공이에요.

---

### Step 3. 진짜 하나인지 증명하고, 공유해서 쓰기 (`Main.java`)

**목표**: 여러 창구가 각자 `getInstance()`를 불러도 **같은 기계**를 쓰는지 확인해요.

**할 일**
1. `getInstance()`를 여러 번 호출해 `w1`, `w2`, `w3`에 담아요.
2. 번갈아 `issue()` 해보고, 번호가 `1, 2, 3, 4…`로 **안 겹치고 이어지는지** 봐요.
3. `w1 == w2`를 출력해 **같은 객체인지**(참조 비교) 증명해요.

<details>
<summary>💡 힌트 보기</summary>

```java
TicketMachine w1 = TicketMachine.getInstance();
TicketMachine w2 = TicketMachine.getInstance();
TicketMachine w3 = TicketMachine.getInstance();

System.out.println(w1.issue());   // 1
System.out.println(w2.issue());   // 2  ← w2도 같은 기계라 이어짐!
System.out.println(w1.issue());   // 3
System.out.println(w3.issue());   // 4
System.out.println(w1 == w2);     // true ← 완전히 같은 객체
```

`==`는 "**같은 객체를 가리키는가**"를 봐요. `true`가 나오면 셋 다 진짜 한 객체라는 증거예요.

</details>

**확인**: 번호가 중복 없이 이어지고, `w1 == w2`가 `true`면 성공이에요.

---

### Step 4. (변형) lazy 초기화 — 필요할 때 만들기 (`Settings.java`)

**목표**: 미리 만들지 않고, **처음 `getInstance()`가 불릴 때** 객체를 만드는 방식을 익혀요.

**할 일**
1. `Settings`의 `instance`를 `final` 없이 `null`로 시작하게 둬요.
2. `getInstance()` 안에서 `instance == null`일 때만 `new`로 만들어요.
3. `theme` 같은 설정값을 두고, 어디서 꺼내든 같은 값이 공유되는지 확인해요.

<details>
<summary>💡 힌트 보기</summary>

```java
class Settings {
    private static Settings instance;          // null로 시작 (미리 안 만듦)
    private String theme = "dark";

    private Settings() {}
    static Settings getInstance() {
        if (instance == null) {                // 처음 호출 때 딱 한 번 생성
            instance = new Settings();
        }
        return instance;
    }
    String getTheme()       { return theme; }
    void   setTheme(String t){ theme = t;    }
}
```

⚠️ **주의**: 두 스레드가 동시에 `if (instance == null)`을 통과하면 객체가 2개 만들어질 수 있어요. 단일 스레드에선 괜찮지만, 멀티스레드 대비는 도전 과제에서 다뤄요.

</details>

**확인**: `Settings.getInstance()`를 두 번 받아 한쪽에서 `setTheme`을 바꾸면 다른 쪽도 바뀌고, `s1 == s2`가 `true`면 성공이에요.

---

### Step 5. 마무리 — Main에서 전체 실행 (`Main.java`)

**목표**: 버그 재현 → 싱글톤 해결 → lazy 변형을 한자리에서 돌려봐요.

**할 일**
1. (1) `NaiveTicketMachine` 두 대로 중복 버그 보여주기.
2. (2) `TicketMachine.getInstance()`로 여러 창구가 공유 + `==` 증명.
3. (3) `Settings.getInstance()`로 lazy 싱글톤 확인.

<details>
<summary>💡 힌트 보기</summary>

```java
public class Main {
    public static void main(String[] args) {
        // 1) 버그: 기계 두 대 → 번호 중복
        NaiveTicketMachine a = new NaiveTicketMachine();
        NaiveTicketMachine b = new NaiveTicketMachine();
        System.out.println(a.issue() + ", " + b.issue());   // 1, 1

        // 2) 해결: 싱글톤은 하나뿐
        TicketMachine w1 = TicketMachine.getInstance();
        TicketMachine w2 = TicketMachine.getInstance();
        System.out.println(w1.issue() + ", " + w2.issue()); // 1, 2
        System.out.println(w1 == w2);                       // true

        // 3) lazy
        Settings s1 = Settings.getInstance();
        s1.setTheme("dark");
        System.out.println(Settings.getInstance().getTheme());  // dark
    }
}
```

(1번 섹션과 똑같은 출력이 되도록 메시지를 맞춰보세요.)

</details>

**확인**: `javac *.java` → `java -Dstdout.encoding=UTF-8 Main` 했을 때 1번 섹션의 출력이 그대로 나오면 **완성입니다!** 🎉

## 6. 학습 체크

- [ ] "객체가 여러 개면 안 되는" 상황을 예로 들 수 있다
- [ ] 생성자를 `private`으로 막으면 왜 `new`가 안 되는지 안다
- [ ] `getInstance()`가 항상 같은 객체를 주는 이유(`static` 보관)를 설명할 수 있다
- [ ] `==`로 "같은 객체"임을 증명할 수 있다
- [ ] eager와 lazy의 차이, 그리고 lazy의 멀티스레드 주의점을 안다

## 7. 최종 완성 체크리스트

- [ ] `TicketMachine`의 생성자가 `private`이다
- [ ] 외부에서 `new TicketMachine()`이 컴파일되지 않는다
- [ ] `getInstance()`로 얻은 객체들이 `==` 비교에서 `true`다
- [ ] 여러 창구가 발급해도 번호가 중복 없이 이어진다
- [ ] `Settings`가 lazy 방식으로 처음 호출 때 만들어진다
- [ ] 1번 섹션과 같은 출력이 나온다

## 8. (선택) 도전 과제

1. `TicketMachine`에 `reset()`(번호를 0으로) 을 추가하고, 어디서 reset해도 모든 창구에 반영되는지 확인해보세요. (공유 상태 체감)
2. 발급된 마지막 번호를 보여주는 `peek()`(올리지 않고 현재값만) 을 추가해보세요.
3. ★ lazy 방식의 멀티스레드 문제를 `synchronized`로 막아보세요. `getInstance()`에 `static synchronized`를 붙이면 어떤 일이 생기는지 조사해보세요.
4. ★ `enum`으로 싱글톤을 만드는 방법(`enum Settings { INSTANCE; … }`)을 조사해보세요. 자바에서 가장 안전한 싱글톤으로 꼽혀요.
5. ★ 싱글톤의 **단점**(테스트하기 어렵다, 전역 상태가 늘어난다)을 한 문단으로 정리해보세요. 지난 시간 배운 IoC/DI가 이 단점을 어떻게 완화하는지도 함께요.
