# JUnit 5 테스트 코드 작성하기 (DAO를 '코드로' 자동 검증하기)

> JUnit 5 테스트 작성법(given-when-then, `@BeforeEach`, `assertEquals`, `assertThrows`, `@Disabled`)을, **메모리 `ProductDao`** 에 직접 테스트를 짜며 손에 익혀요.
> 마지막엔 **스프링 주입(`@SpringJUnitConfig` + `@Autowired`)** 버전까지 올라가요.
> **아래 Step을 순서대로 따라가면, 마지막 Step에서 통과하는 테스트 묶음이 완성됩니다.**
>
> 💡 각 Step의 **힌트는 접혀 있어요.** "이 동작이 맞는지 코드로 어떻게 확인하지?"를 먼저 떠올리고, 막히면 펼치세요.

---

## 0. 먼저 알아둘 점

- 이 과제는 *테스트 작성법* 자체에 집중하려고, **DB 없이 메모리(HashMap)로 도는 `ProductDao`** 를 검증 대상으로 써요. DB에 붙는 DAO를 테스트할 때와 **테스트를 짜는 방식은 동일**해요. DB 준비 부담만 없앤 거예요.
- 메모리로 단순화하면서 예외 종류만 살짝 달라요. (작성법은 같아요.)

  | DB 기반이라면 | 이 과제(메모리) | 상황 |
  |---|---|---|
  | `SQLException` (중복키) | `IllegalStateException` | 같은 id 두 번 add |
  | `SQLException` (행 없음) | `NoSuchElementException` | 없는 id로 get |

- **JUnit 5는 프로젝트에 이미 있다고 가정**해요. (Gradle이라면 `build.gradle`에 `testImplementation 'org.junit.jupiter:junit-jupiter'`가 있고, `test { useJUnitPlatform() }` 설정이 있어요.) 실행은 IDE의 ▶ 버튼이나 `./gradlew test`.
- `assertThrows`에 넘기는 코드는 **익명 클래스(`Executable`)** 로 만들어요.
- 스프링 주입 버전은 **Step 6(심화)** 에서 다뤄요.

---

## 1. 무엇을 만드나요?

아래 **검증 대상 코드**(`Product`, `ProductDao`)는 제공돼요. 여러분은 이걸 검증하는 `ProductDaoTest`를 작성해요.

```java
// Product.java (제공)
public class Product {
    private String id;
    private String name;
    private int price;
    public String getId() { return id; }            public void setId(String id) { this.id = id; }
    public String getName() { return name; }         public void setName(String name) { this.name = name; }
    public int getPrice() { return price; }          public void setPrice(int price) { this.price = price; }
}
```
```java
// ProductDao.java (제공) — 메모리 저장. DB 대신 HashMap을 씀.
import java.util.HashMap;
import java.util.NoSuchElementException;

public class ProductDao {
    private HashMap<String, Product> store = new HashMap<>();

    public void add(Product product) {
        if (store.containsKey(product.getId()))
            throw new IllegalStateException("이미 존재하는 id: " + product.getId());
        store.put(product.getId(), product);
    }
    public Product get(String id) {
        Product found = store.get(id);
        if (found == null) throw new NoSuchElementException("없는 id: " + id);
        return found;
    }
    public int getCount() { return store.size(); }
    public void deleteAll() { store.clear(); }
}
```

**완성되면** 테스트를 실행했을 때 이런 모습이에요(IDE에서 초록 체크).

```
ProductDaoTest
  ✓ add()                     통과
  ✓ get()                     통과
  ✓ add_중복_id_예외()         통과
  ✓ get_없는_id_예외()         통과
  ⊘ 일부러_실패하는_테스트()    건너뜀(@Disabled)
```

## 3. 핵심 개념

### (1) 왜 테스트 코드를 짜나요?
`main` + `System.out.println`으로 결과를 **눈으로** 확인하는 대신, **`assert`(단언)로 코드가 자동 검증**하게 해요. 장점은 두 가지예요: ① 사람이 매번 안 봐도 되고, ② DAO가 늘어나도 테스트만 추가하면 **한 번에 자동 실행**돼요.

### (2) given-when-then (테스트의 3박자)
```
given (준비) : 테스트할 상황을 만든다           예) 비어 있는 상태에서 시작
when  (실행) : 검증할 동작을 실제로 한 번 한다   예) add(product)
then  (검증) : 결과가 기대와 같은지 단언한다     예) getCount()가 1이어야
```

### (3) `@BeforeEach`로 '항상 같은 출발점'
`@BeforeEach`가 붙은 메서드는 **각 `@Test` 직전마다 매번** 실행돼요. 여기서 `deleteAll()`로 **항상 0건에서 시작**하게 하면, 테스트끼리 데이터가 안 섞이고 **몇 번 돌려도 결과가 같아요(반복 가능)**.

### (4) `assertEquals` / `assertThrows`
```java
assertEquals(기대값, 실제값);          // 둘이 다르면 실패 (순서 주의: 기대 먼저!)
assertThrows(기대예외.class, 실행코드);  // 실행코드가 그 예외를 던지면 성공
```
`assertThrows`의 '실행코드'는 `Executable` 인터페이스의 `execute()`로 넘겨요. **익명 클래스**로 그 자리에서 구현해 넘기는 방식을 써요.

### (5) `@Disabled`와 실패 메시지
`@Disabled`는 그 테스트를 **평소엔 실행 안 함**으로 막아둬요(깨진 테스트가 전체 빌드를 막지 않게). 일부러 틀린 기대값을 넣고 잠시 활성화하면 JUnit이 `expected: <2> but was: <1>` 처럼 **기대값과 실제값**을 알려줘요 — 실패 메시지 읽는 법을 익히는 거예요.

```
[기억법] given 준비 → when 한 번 실행 → then 단언 / 매 테스트 전 @BeforeEach로 리셋
```

## 5. Step by Step

### Step 1. 테스트 클래스 + 매번 같은 출발점 만들기 (`ProductDaoTest.java`)

**목표**: 테스트 클래스를 만들고, `@BeforeEach`로 **매 테스트 전에 0건**이 되게 해요.

**할 일**
1. `ProductDaoTest` 클래스를 만들고, `private ProductDao dao = new ProductDao();` 필드를 둬요.
2. `@BeforeEach` 메서드 `setUp()`에서 `dao.deleteAll()`을 호출해요.
3. 테스트마다 상품을 만들 도우미 `newProduct(id, name, price)`도 만들어요.

<details>
<summary>💡 힌트 보기</summary>

```java
import org.junit.jupiter.api.BeforeEach;
import org.junit.jupiter.api.Test;
import org.junit.jupiter.api.Disabled;
import org.junit.jupiter.api.function.Executable;
import java.util.NoSuchElementException;
import static org.junit.jupiter.api.Assertions.*;

class ProductDaoTest {

    private ProductDao dao = new ProductDao();

    @BeforeEach
    void setUp() {
        dao.deleteAll();          // 매 테스트 직전 0건으로 리셋
    }

    private Product newProduct(String id, String name, int price) {
        Product p = new Product();
        p.setId(id); p.setName(name); p.setPrice(price);
        return p;
    }
}
```

`import static ... Assertions.*` 덕분에 `assertEquals`를 클래스 이름 없이 바로 써요. 핵심은 `@BeforeEach`가 **자동으로 매번** 불린다는 점이에요.

</details>

**확인**: 클래스가 컴파일되고, 다음 Step의 테스트들이 항상 0건에서 시작할 준비가 되면 통과예요.

---

### Step 2. add() 테스트 — given-when-then (`ProductDaoTest.java`)

**목표**: `add` 후 **개수가 1 늘었는지**로 add의 정상 동작을 검증해요. (get에 기대지 않고)

**할 일**
1. `@Test void add()`를 만들어요.
2. given: 시작이 0건인지 `assertEquals(0, dao.getCount())`로 확인.
3. when: `dao.add(newProduct(...))`.
4. then: `assertEquals(1, dao.getCount())`.

<details>
<summary>💡 힌트 보기</summary>

```java
@Test
void add() {
    // given : 비어 있는 상태(0건)
    assertEquals(0, dao.getCount());
    Product product = newProduct("p1", "연필", 500);

    // when : 저장
    dao.add(product);

    // then : 1건이 되어야 한다  (기대값 먼저, 실제값 나중!)
    assertEquals(1, dao.getCount());
}
```

`assertEquals(1, dao.getCount())`처럼 **기대값(1)을 앞**에 써요. 순서가 바뀌면 실패 메시지의 expected/actual이 거꾸로 나와 헷갈려요.

</details>

**확인**: `add()` 테스트가 초록불(통과)이면 성공이에요.

---

### Step 3. get() 테스트 — 조회 결과의 정확성 (`ProductDaoTest.java`)

**목표**: `get(id)`이 **저장한 값과 똑같이** 돌려주는지 검증해요.

**할 일**
1. given: 조회할 상품을 **먼저 `add`로 준비**해요. (여기서 add는 검증 대상이 아니라 *사전 준비*예요.)
2. when: `dao.get(id)`로 조회.
3. then: 이름과 가격이 저장한 값과 같은지 각각 `assertEquals`로 확인.

<details>
<summary>💡 힌트 보기</summary>

```java
@Test
void get() {
    // given : 조회 대상 미리 저장
    Product product = newProduct("p1", "연필", 500);
    dao.add(product);

    // when : id로 조회
    Product found = dao.get("p1");

    // then : 모든 필드가 같아야 한다
    assertEquals(product.getName(),  found.getName());
    assertEquals(product.getPrice(), found.getPrice());
}
```

조회를 검증하려면 데이터가 먼저 있어야 하니, given에서 `add`로 깔아두는 게 자연스러워요.

</details>

**확인**: `get()` 테스트가 통과하면 성공이에요.

---

### Step 4. 예외 테스트 — assertThrows + 익명 클래스 (`ProductDaoTest.java`)

**목표**: '예외가 나는 것'이 정상인 상황을, **익명 `Executable`** 로 검증해요.

**할 일**
1. `add_중복_id_예외()`: 같은 상품을 두 번 `add` 하면 `IllegalStateException`이 나야 해요.
2. `get_없는_id_예외()`: 없는 id로 `get` 하면 `NoSuchElementException`이 나야 해요.
3. 두 경우 모두 `assertThrows(예외.class, action)`을 쓰고, `action`은 **익명 클래스**로 만들어요.

<details>
<summary>💡 힌트 보기</summary>

```java
@Test
void add_중복_id_예외() {
    final Product product = newProduct("dup", "지우개", 300);
    dao.add(product);                       // 첫 저장은 정상

    Executable action = new Executable() {  // 예외가 날 코드를 익명 클래스로
        @Override
        public void execute() {
            dao.add(product);               // 같은 id 재저장 → 예외!
        }
    };
    assertThrows(IllegalStateException.class, action);
}

@Test
void get_없는_id_예외() {
    Executable action = new Executable() {
        @Override
        public void execute() {
            dao.get("없는_id");             // 행 없음 → 예외!
        }
    };
    assertThrows(NoSuchElementException.class, action);
}
```

익명 클래스에서 바깥 변수(`product`)를 쓰려면 `final`이 필요해요. 이 익명 클래스를 람다로 줄이면 `assertThrows(IllegalStateException.class, () -> dao.add(product));` 한 줄이 돼요.

</details>

**확인**: 두 예외 테스트가 모두 통과하면 성공이에요. (예외 *타입*까지 맞아야 통과해요.)

---

### Step 5. 실패 테스트 체험 — @Disabled & 실패 메시지 (`ProductDaoTest.java`)

**목표**: 테스트가 **실패하면 어떻게 보이는지** 직접 봐요.

**할 일**
1. `@Disabled`와 `@Test`를 단 `일부러_실패하는_테스트()`를 만들어요.
2. 상품 1개를 add 한 뒤, **일부러 2건을 기대**하는 단언을 넣어요.
3. `@Disabled`를 잠시 주석 처리하고 실행해, 실패 메시지를 직접 읽어봐요. 확인했으면 다시 `@Disabled`를 켜둬요.

<details>
<summary>💡 힌트 보기</summary>

```java
@Disabled("일부러 틀린 기대값을 넣은 학습용 실패 예제")
@Test
void 일부러_실패하는_테스트() {
    dao.add(newProduct("fail_demo", "공책", 1000));
    assertEquals(2, dao.getCount());   // 실제는 1건인데 2 기대 → 실패
}
```

`@Disabled`를 잠깐 끄고 돌리면 `expected: <2> but was: <1>` 같은 메시지가 떠요. "기대값 2, 실제값 1"을 알려주는 거예요. 평소엔 다시 `@Disabled`로 막아둬서 빌드가 안 깨지게 해요.

</details>

**확인**: 활성화 시 빨간 실패와 expected/actual 메시지를 확인하고, 다시 `@Disabled`로 꺼두면 성공이에요.

### Step 6. (심화) 스프링 컨테이너로 주입받기 (`AppConfig.java`, `ProductDaoTest.java`)

**목표**: `new ProductDao()`를 직접 만드는 대신, **스프링이 빈을 주입**하게 바꿔요. (`@SpringJUnitConfig` + `@Autowired`)

**할 일**
1. `@Configuration` 클래스 `AppConfig`를 만들고, `@Bean`으로 `ProductDao`를 등록해요.
2. 테스트 클래스에 `@SpringJUnitConfig(AppConfig.class)`를 붙여요.
3. 필드를 `@Autowired private ProductDao dao;`로 바꿔요. (직접 `new` 제거)
4. `@BeforeEach`의 `deleteAll()`은 그대로 둬요 — 빈은 싱글톤이라 더더욱 매번 리셋이 필요해요.

<details>
<summary>💡 힌트 보기</summary>

```java
// AppConfig.java
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;

@Configuration
public class AppConfig {
    @Bean
    public ProductDao productDao() {
        return new ProductDao();
    }
}
```
```java
// ProductDaoTest.java 상단만 변경
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.test.context.junit.jupiter.SpringJUnitConfig;

@SpringJUnitConfig(AppConfig.class)
class ProductDaoTest {

    @Autowired
    private ProductDao dao;        // 직접 new 하지 않고 주입받음

    @BeforeEach
    void setUp() { dao.deleteAll(); }
    // ... 나머지 테스트는 그대로
}
```

`@SpringJUnitConfig(AppConfig.class)`는 사실 두 개를 합친 거예요: ① `@ExtendWith(SpringExtension.class)`(JUnit5에 스프링 기능 끼우기) + ② `@ContextConfiguration(classes = AppConfig.class)`(어떤 설정으로 컨테이너를 띄울지). 그래서 테스트가 **스프링 컨테이너 안에서** 돌고, `@Autowired` 필드에 타입이 맞는 빈이 자동으로 꽂혀요. (이 애너테이션이 없으면 컨테이너가 없어 `dao`가 `null` → `NullPointerException`!)

⚠️ 이 Step은 프로젝트에 `spring-context`, `spring-test` 의존성이 있어야 동작해요. 없으면 Step 5까지가 과제 완성이고, 이건 참고용으로 읽어도 돼요.

</details>

**확인**: `@Autowired`로 주입받은 `dao`로 Step 2~4 테스트가 그대로 통과하면 성공이에요.

### Step 마지막. 마무리 점검 (완성!)

**점검 항목**
- [ ] `@BeforeEach`로 매 테스트가 0건에서 시작한다

- [ ] `@Disabled` 실패 예제의 메시지를 한 번 봤다

## 6. 학습 체크

- [ ] 왜 `println` 대신 `assert`로 검증하는지 설명할 수 있다
- [ ] given-when-then 3박자를 직접 나눌 수 있다
- [ ] `@BeforeEach`가 언제·왜 매번 실행되는지 안다
- [ ] `assertEquals`의 인자 순서(기대, 실제)를 안다
