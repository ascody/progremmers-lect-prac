# 제어의 역전(IoC) 익히기 (제어권을 내 손에서 바깥으로 넘기기)

> "제어의 역전(Inversion of Control)"을, **제어권이 내 코드에서 → 바깥(컨테이너/프레임워크)으로 넘어가는 과정**을 단계별로 만들어보며 익혀요.
> 커피 머신 예제 하나로 *직접 제어 → DI → IoC 컨테이너 → 흐름의 역전*까지 사다리를 올라가요.
> **아래 Step을 순서대로 따라가면, 마지막 Step에서 IoC가 동작하는 프로그램이 완성됩니다.**
>
> 💡 각 Step의 **힌트는 접혀 있어요.** 먼저 "내가 제어하던 걸 어떻게 바깥으로 넘기지?"를 스스로 고민하고, 막히면 펼치세요.

---

## 0. 먼저 알아둘 점

지난 시간 **DIP(의존관계 역전 원칙)** 와 헷갈리기 쉬워서, 관계부터 정리하고 갈게요.

| 용어 | 정체 | 한 줄 |
|------|------|------|
| DIP | **원칙** | "구체 클래스 말고 추상에 의존해라" |
| DI(의존성 주입) | **기법** | "필요한 객체를 직접 `new` 말고 주입받아라" |
| IoC(제어의 역전) | **더 큰 개념** | "내가 쥐고 있던 **제어권**을 바깥에 넘겨라" |

즉 **DI는 IoC를 실현하는 가장 흔한 방법**이에요. 사실 지난 시간 `NotificationService`가 `MessageSender`를 생성자로 주입받은 것도 이미 IoC의 한 사례였어요. 이번엔 그걸 더 끝까지 밀어붙여요.

그리고 IoC에는 **두 얼굴**이 있어요. 이 가이드는 둘 다 다뤄요.
- **(가) 객체 생성·조립의 제어 역전** → Step 1~3 (커피 머신 + 컨테이너)
- **(나) 실행 흐름의 제어 역전 = 헐리우드 원칙** → Step 4 (버튼 + 콜백)

> 참고: 실무에선 Spring 같은 프레임워크가 이 "컨테이너"를 자동으로 해줘요. 여기선 원리를 손으로 느끼려고 **컨테이너를 직접 작게** 만들어봐요.

## 1. 무엇을 만드나요?

제어권이 단계별로 바깥으로 넘어가는 걸 보여주는 `Main`을 완성해요. 다 만들면 이렇게 나와요.

```
===== 2. DI: 제어를 바깥(main)으로 =====
콜롬비아 원두로 커피를 내립니다
에티오피아 원두로 커피를 내립니다

===== 3. IoC 컨테이너: 조립까지 위임 =====
콜롬비아 원두로 커피를 내립니다

===== 4. 헐리우드 원칙: 흐름의 역전 =====
[시스템] 버튼이 눌렸습니다
내 코드 실행: 좋아요!
```

(Step 1은 "IoC 적용 전"의 출발점이라 위 출력엔 없어요. 거기서 시작해 한 칸씩 개선해 나가요.)

## 3. 핵심 개념

### (1) "제어"가 뭔데 역전된다는 거예요?
프로그램에는 두 종류의 제어권이 있어요.
- **무엇을 만들고 연결할지** (객체 생성·조립)
- **언제 무엇을 실행할지** (실행 흐름)

보통은 *내 코드가* 이 둘을 쥐고 있어요. 내가 `new`로 객체를 만들고, 내가 메서드를 호출하죠. **IoC는 이 제어권을 바깥(컨테이너/프레임워크)에 넘기는 것**이에요.

```
[IoC 전]  내 코드 ──new──▶ 의존 객체     (내가 만들고, 내가 호출)
[IoC 후]  컨테이너 ──조립──▶ 객체 ──넘겨줌──▶ 내 코드   (바깥이 만들어 건네줌)
                프레임워크 ──호출──▶ 내 코드   (바깥이 내 코드를 부름)
```

### (2) 첫 번째 얼굴: 생성·조립의 역전 (DI → 컨테이너)
`CoffeeMaker`가 원두를 **스스로 `new`** 하면, 원두를 바꿀 때마다 `CoffeeMaker`를 수정해야 해요. → 생성자로 **주입**받으면 "무엇을 쓸지"의 제어권이 바깥으로. → 더 나아가 **컨테이너**가 조립을 전담하면, 내 코드(`main`)는 `new`조차 안 하고 "완성품 줘"라고 요청만 해요.

### (3) 두 번째 얼굴: 흐름의 역전 (헐리우드 원칙)
> "Don't call us, we'll call you" (먼저 연락하지 마, 우리가 부를게)

내가 라이브러리를 호출하는 게 보통이지만, 거꾸로 **프레임워크가 내 코드를 호출**하기도 해요. 버튼을 누르는 *시점*은 버튼(시스템)이 정하고, 그때 내가 등록해둔 코드(리스너)를 불러줘요. **언제 실행될지의 제어권이 나 → 프레임워크로 넘어간** 거예요.

```
[기억법] IoC = "내가 하던 걸, 바깥이 하게 한다"
  · 객체 만들기  → 컨테이너가
  · 코드 부르기  → 프레임워크가
```

## 5. Step by Step

### Step 1. (출발점) 직접 제어의 한계 느끼기 (`CoffeeMaker.java`)

**목표**: 객체를 스스로 `new` 하면 왜 불편한지 **직접 겪어봐요.** (이게 IoC를 적용하기 *전* 상태예요.)

**할 일**
1. `CoffeeMaker`가 내부에서 원두를 **직접** `new ColombiaBean()` 하도록 짜요.
2. 그런 다음 "에티오피아 원두로 바꾸려면 어디를 고쳐야 하지?"를 스스로 답해봐요.

<details>
<summary>💡 힌트 보기</summary>

```java
// ❌ IoC 적용 전: CoffeeMaker가 '무슨 원두 쓸지'까지 스스로 결정 = 제어권이 내 안에
class CoffeeMaker {
    private Bean bean = new ColombiaBean();   // 여기서 못 박힘
    void brew() {
        System.out.println(bean.name() + "로 커피를 내립니다");
    }
}
```

원두를 바꾸려면 **`CoffeeMaker` 클래스 자체를 수정**해야 해요. `CoffeeMaker`가 너무 많은 걸 *제어*하고 있는 거예요. (`Bean` 인터페이스는 Step 2 힌트에 있어요.)

</details>

**확인**: "원두를 바꾸려면 `CoffeeMaker`를 고쳐야 한다"는 불편함을 말로 설명할 수 있으면 통과예요. 이 불편함을 Step 2~3에서 없애요.

---

### Step 2. DI — "무엇을 쓸지"의 제어권을 바깥으로 (`Bean.java`, `CoffeeMaker.java`)

**목표**: 원두를 직접 `new` 하지 말고 **주입**받게 해서, 선택 제어권을 `CoffeeMaker` 밖으로 빼요.

**할 일**
1. `Bean` 인터페이스(`String name();`)와 구현 `ColombiaBean`, `EthiopiaBean`을 만들어요.
2. `CoffeeMaker`가 `Bean`을 **생성자로 주입**받게 고쳐요. (내부에서 `new` 금지!)

<details>
<summary>💡 힌트 보기</summary>

```java
// Bean.java
interface Bean { String name(); }
class ColombiaBean  implements Bean { public String name() { return "콜롬비아 원두"; } }
class EthiopiaBean  implements Bean { public String name() { return "에티오피아 원두"; } }

// CoffeeMaker.java
class CoffeeMaker {
    private Bean bean;
    CoffeeMaker(Bean bean) { this.bean = bean; }   // 주입! 스스로 안 정함
    void brew() { System.out.println(bean.name() + "로 커피를 내립니다"); }
}
```

이제 "어떤 원두?"를 `CoffeeMaker`가 아니라 **부르는 쪽(main)** 이 정해요. `new CoffeeMaker(new EthiopiaBean())`처럼요. 제어권이 한 칸 바깥으로 나갔어요.

</details>

**확인**: `CoffeeMaker` 안에 `new ColombiaBean()` 같은 코드가 **하나도 없고**, 콜롬비아·에티오피아를 둘 다 주입해 서로 다른 출력이 나오면 성공이에요.

---

### Step 3. IoC 컨테이너 — 조립까지 위임하기 (`CoffeeContainer.java`)

**목표**: `main`이 `new`조차 하지 않도록, **객체 생성·조립을 전담하는 컨테이너**를 만들어요.

**할 일**
1. `CoffeeContainer` 클래스를 만들어요.
2. 그 안에 `CoffeeMaker getCoffeeMaker()` 메서드를 두고, **원두 생성 + 머신 조립**을 거기서 다 하게 해요.
3. `main`은 컨테이너에게 "완성된 머신 줘"라고 **요청만** 하게 해요.

<details>
<summary>💡 힌트 보기</summary>

```java
class CoffeeContainer {
    CoffeeMaker getCoffeeMaker() {
        Bean bean = new ColombiaBean();   // 무엇을 쓸지 → 컨테이너가 결정
        return new CoffeeMaker(bean);     // 조립 → 컨테이너가 책임
    }
}
```

사용하는 쪽은 이렇게 바뀌어요.
```java
CoffeeContainer container = new CoffeeContainer();
CoffeeMaker maker = container.getCoffeeMaker();   // main은 new를 안 함!
maker.brew();
```

**제어권이 컨테이너로 완전히 넘어간** 상태예요. (Spring은 이 컨테이너를 자동으로 만들어줄 뿐, 원리는 똑같아요.)

</details>

**확인**: `main`에 `new CoffeeMaker(...)`나 `new ColombiaBean()`이 없고, `container.getCoffeeMaker()`로만 머신을 얻으면 성공이에요.

---

### Step 4. 헐리우드 원칙 — 흐름의 제어를 넘기기 (`Hollywood.java`)

**목표**: "내가 호출"이 아니라 **"프레임워크가 내 코드를 호출"** 하는 구조를 만들어요. (실행 흐름의 역전)

**할 일**
1. `ClickListener` 인터페이스(`void onClick();`)를 만들어요.
2. `Button`이 리스너를 `setListener`로 받아두고, `press()` 안에서 **`listener.onClick()`을 호출**하게 해요.
3. 내가 끼워 넣을 코드 `LikeAction`(`ClickListener` 구현)을 만들어요.

<details>
<summary>💡 힌트 보기</summary>

```java
interface ClickListener { void onClick(); }

class Button {
    private ClickListener listener;
    void setListener(ClickListener l) { this.listener = l; }
    void press() {
        System.out.println("[시스템] 버튼이 눌렸습니다");
        listener.onClick();          // ← 내가 부르는 게 아니라, Button이 내 코드를 부름
    }
}

class LikeAction implements ClickListener {
    public void onClick() { System.out.println("내 코드 실행: 좋아요!"); }
}
```

핵심은 **`onClick()`을 내가 직접 호출하지 않는다**는 점이에요. 나는 `setListener`로 *끼워두기만* 하고, 부르는 시점은 `Button`이 정해요. 그게 "흐름 제어가 역전됐다"는 뜻이에요.

</details>

**확인**: `main`에서 `onClick()`을 직접 호출하지 않고, `button.press()`만 했는데 내 `LikeAction`이 실행되면 성공이에요.

---

### Step 5. 마무리 — Main에서 단계별로 실행 (`Main.java`)

**목표**: 지금까지 만든 걸 한자리에서 돌려, 제어권이 단계별로 넘어가는 걸 눈으로 확인해요.

**할 일**
1. (2) DI: 콜롬비아·에티오피아를 각각 주입해 `brew()`.
2. (3) 컨테이너: `getCoffeeMaker()`로 받아 `brew()`.
3. (4) 헐리우드: `Button`에 `LikeAction`을 등록하고 `press()`.

<details>
<summary>💡 힌트 보기</summary>

```java
public class Main {
    public static void main(String[] args) {
        System.out.println("===== 2. DI =====");
        new CoffeeMaker(new ColombiaBean()).brew();
        new CoffeeMaker(new EthiopiaBean()).brew();

        System.out.println("\n===== 3. IoC 컨테이너 =====");
        CoffeeMaker maker = new CoffeeContainer().getCoffeeMaker();
        maker.brew();

        System.out.println("\n===== 4. 헐리우드 원칙 =====");
        Button button = new Button();
        button.setListener(new LikeAction());   // 끼워두기만
        button.press();                         // 호출은 Button이
    }
}
```

</details>

**확인**: `javac *.java` → `java -Dstdout.encoding=UTF-8 Main` 했을 때 1번 섹션의 출력이 그대로 나오면 **완성입니다!** 🎉
