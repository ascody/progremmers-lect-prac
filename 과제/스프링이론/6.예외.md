# 예외 처리 3전략 + 파일 로깅 만들기 (재시도 복구 & 바탕화면 로그 쌓기)

> 예외를 다루는 세 가지 방법(**복구 / 회피 / 전환**)을 실제 코드로 익히고, 그 과정을 **바탕화면의 로그 파일에 누적 기록**해요.
> 특히 **재시도(retry)로 일시적 오류를 복구**하고, 각 시도를 로그로 남기는 게 이번 과제의 중심이에요.
> **아래 Step을 순서대로 따라가면, 마지막 Step에서 바탕화면에 로그가 쌓이는 프로그램이 완성됩니다.**
>
> 💡 각 Step의 **힌트는 접혀 있어요.** "이 예외를 어떻게 처리하는 게 맞지?"를 먼저 고민하고, 막히면 펼치세요.

---

## 0. 먼저 알아둘 점

- 예외 처리의 큰 원칙은 **"모든 예외는 복구되든지, 아니면 분명히 통보되어야 한다"** 예요. 그냥 `catch`로 잡고 아무것도 안 하는 **'예외 블랙홀'은 금지**예요.
- 이번 과제는 이 원칙을 지키는 세 가지 방법을 코드로 만들어요: ① **복구**(재시도) ② **회피**(호출자에게 넘김) ③ **전환**(의미 있는 예외로 바꿈).
- **로그는 파일로 남겨요.** 바탕화면에 `app-logs` 폴더를 만들고, 그 안 `app.log`에 한 줄씩 **이어 써요(append)**. 프로그램을 다시 돌려도 지워지지 않고 **쌓여요**.
- ⚠️ **바탕화면 경로 주의**: 보통 `사용자홈/Desktop`이에요. 그런데 한글 윈도우·OneDrive 환경에선 위치가 다르거나(예: `OneDrive/바탕 화면`) 리디렉션될 수 있어요. **폴더가 안 보이면 당황 말고**, `logger.getLogFilePath()`로 **실제 저장 경로를 출력**해 확인하세요.
- 파일 입출력엔 `try-with-resources`(자동 close)와 체크 예외(`IOException`, `SQLException`)가 나와요. 예외 테마와 자연스럽게 이어져요.

## 1. 무엇을 만드나요?

세 전략을 실행하고, 그 결과를 콘솔과 **로그 파일** 양쪽에 남기는 프로그램을 완성해요.

**콘솔 출력**
```
===== 1) 예외 복구: 재시도 (3번째에 성공) =====
최종 결과: 데이터-OK

===== 2) 예외 복구 실패: 재시도 모두 실패 -> 통보 =====
실패 통보: 재시도 3회 모두 실패했습니다.

===== 3) 예외 전환: 아이디 중복 -> 의미 있는 예외 =====
잡힘: 이미 존재하는 아이디입니다: kim
원인 보존: java.sql.SQLException: Duplicate entry

===== 로그 파일 위치 =====
C:\Users\사용자\Desktop\app-logs\app.log
```

**바탕화면 `app-logs/app.log` 내용 (누적됨)**
```
2026-06-24 08:43:15 [WARN] 1번째 시도 실패: 일시적 연결 오류 (호출 1)
2026-06-24 08:43:15 [WARN] 2번째 시도 실패: 일시적 연결 오류 (호출 2)
2026-06-24 08:43:15 [INFO] 3번째 시도 성공: 데이터-OK
2026-06-24 08:43:15 [WARN] 1번째 시도 실패: 일시적 연결 오류 (호출 1)
2026-06-24 08:43:15 [WARN] 2번째 시도 실패: 일시적 연결 오류 (호출 2)
2026-06-24 08:43:15 [WARN] 3번째 시도 실패: 일시적 연결 오류 (호출 3)
2026-06-24 08:43:15 [ERROR] 재시도 3회 모두 실패
2026-06-24 08:43:15 [ERROR] 아이디 중복: kim
```

핵심은 **재시도로 복구되는 과정(WARN→WARN→INFO)** 과, 그게 **파일에 그대로 쌓이는** 모습이에요.

## 2. 학습 목표

| 개념 | 어디서 배우나 |
|------|------|
| 파일 로거: 폴더 생성 + append 누적 | Step 1 (`FileLogger.java`) |
| 예외 복구: 재시도(retry)와 통보 | Step 2 (`DataService.java`) |
| 예외 회피: throws / 다시 던지기 | Step 3 (`DataService.java`) |
| 예외 전환: 의미 있는 예외 + 원인 보존 | Step 4 (`DataService.java`) |
| 전체 실행 + 바탕화면 로그 확인 | Step 5 (`Main.java`) |

## 3. 핵심 개념

### (1) 예외 처리 3전략 한눈에
| 전략 | 한 줄 | 핵심 |
|------|------|------|
| 복구(recovery) | 문제를 해결해 **정상 흐름으로 되돌림** | 재시도, 대체값. (catch 후 무시는 복구 아님!) |
| 회피(avoidance) | **호출한 쪽으로 넘김** | `throws` 또는 잡았다가 `throw e` |
| 전환(translation) | **더 적절한 예외로 바꿔** 던짐 | 원인(cause)을 꼭 담는다 |

### (2) 예외 복구 = 재시도 (이번 과제의 중심)
일시적 오류(네트워크 끊김 등)는 **다시 해보면 성공**할 수 있어요. 정해진 횟수만큼 재시도하고, 성공하면 **정상값을 반환(복구)**, 끝까지 실패하면 **분명히 통보(예외 던지기)** 해요.
```
시도1 실패(WARN) → 시도2 실패(WARN) → 시도3 성공(INFO) → 정상 반환  ← 복구!
시도1~3 모두 실패 → ERROR 기록 → 예외 throw                       ← 통보!
```

### (3) 예외 전환 = 의미 부여 + 원인 보존
저수준의 모호한 `SQLException`을, 호출자가 알아듣는 의미 있는 예외로 바꿔요. 이때 **원래 예외를 cause로 반드시 담아야** 스택트레이스가 보존돼요.
```java
catch (SQLException e) {
    throw new DuplicateUserIdException(id, e);  // ← e(원인)를 함께 전달
}
```

### (4) 파일 로깅 3요소
```
① 폴더 생성    logDir.mkdirs();                    // 없으면 만든다
② 이어쓰기     new FileWriter(logFile, true)       // true = append(누적)
③ 자동 닫기    try (FileWriter fw = ...) { ... }    // try-with-resources
```

```
[기억법] 복구=되돌리기 / 회피=넘기기 / 전환=바꾸기(원인 보존) · 로그는 mkdirs + append
```

## 4. 파일 구조

| 파일 | 역할 |
|------|------|
| `FileLogger.java` | 바탕화면 폴더에 로그를 누적 기록 |
| `FlakyService.java` | (제공) 정해진 횟수만큼 실패하는 가짜 서비스 — 재시도 시연용 |
| `DataService.java` | 3가지 예외 전략 + 사용자 정의 예외 |
| `Main.java` | 시나리오를 실행하는 진입점 |

> 실행: `javac *.java` → `java -Dstdout.encoding=UTF-8 Main`

## 5. Step by Step

### Step 1. 바탕화면에 로그를 쌓는 FileLogger 만들기 (`FileLogger.java`)

**목표**: 바탕화면에 `app-logs` 폴더를 만들고, `app.log`에 **이어 쓰는** 로거를 만들어요.

**할 일**
1. 생성자에서 `사용자홈/Desktop/app-logs/app.log` 경로를 잡아요. (`System.getProperty("user.home")`)
2. `log(level, message)`에서 ① 폴더가 없으면 `mkdirs()`로 만들고, ② "시간 [레벨] 메시지"를 **append 모드**로 한 줄 써요.
3. 실제 저장 경로를 알려주는 `getLogFilePath()`도 만들어요.

<details>
<summary>💡 힌트 보기</summary>

```java
import java.io.File;
import java.io.FileWriter;
import java.io.IOException;
import java.time.LocalDateTime;
import java.time.format.DateTimeFormatter;

class FileLogger {
    private final File logDir;
    private final File logFile;
    private static final DateTimeFormatter FMT = DateTimeFormatter.ofPattern("yyyy-MM-dd HH:mm:ss");

    FileLogger() {
        String home = System.getProperty("user.home");
        this.logDir  = new File(home, "Desktop/app-logs");  // 바탕화면/app-logs
        this.logFile = new File(logDir, "app.log");
    }

    void log(String level, String message) {
        if (!logDir.exists()) logDir.mkdirs();              // ① 폴더 생성
        String line = LocalDateTime.now().format(FMT) + " [" + level + "] " + message + System.lineSeparator();
        try (FileWriter fw = new FileWriter(logFile, true)) {  // ② true = 이어쓰기
            fw.write(line);
        } catch (IOException e) {
            System.out.println("로그 기록 실패: " + e.getMessage());
        }
    }

    String getLogFilePath() { return logFile.getAbsolutePath(); }
}
```

`FileWriter(logFile, true)`의 `true`가 핵심이에요. 빼면 매번 **덮어써서** 마지막 줄만 남아요. `try (...)`로 감싸면 끝날 때 파일이 자동으로 닫혀요.

</details>

**확인**: `new FileLogger().log("INFO", "테스트")`를 한 번 호출했을 때, 바탕화면에 `app-logs/app.log`가 생기고 그 줄이 들어 있으면 성공이에요. (안 보이면 `getLogFilePath()`로 경로 확인!)

### Step 2. 예외 복구 — 재시도하고 로그 남기기 (`FlakyService.java`, `DataService.java`)

**목표**: 일시적 오류를 **정해진 횟수만큼 재시도**하고, 성공하면 복구·끝까지 실패하면 통보해요. 각 시도는 로그로 남겨요.

`FlakyService`(제공)는 `failTimes`번 실패한 뒤 성공해요.
```java
// FlakyService.java (제공)
class FlakyService {
    private final int failTimes;
    private int callCount = 0;
    FlakyService(int failTimes) { this.failTimes = failTimes; }
    String fetch() throws java.sql.SQLException {
        callCount++;
        if (callCount <= failTimes)
            throw new java.sql.SQLException("일시적 연결 오류 (호출 " + callCount + ")");
        return "데이터-OK";
    }
}
```

**할 일**
1. `DataService`에 `FileLogger`를 필드로 받아요(생성자 주입).
2. `fetchWithRetry(FlakyService flaky)`를 만들어, 최대 3번 반복하며 `flaky.fetch()`를 시도해요.
3. 성공하면 `INFO` 로그를 남기고 **결과를 반환**해요(복구).
4. 실패하면 `WARN` 로그를 남기고 다시 시도해요.
5. 3번 다 실패하면 `ERROR` 로그를 남기고 `RuntimeException`을 던져요(통보).

<details>
<summary>💡 힌트 보기</summary>

```java
import java.sql.SQLException;

class DataService {
    private final FileLogger logger;
    DataService(FileLogger logger) { this.logger = logger; }

    String fetchWithRetry(FlakyService flaky) {
        int maxRetry = 3;
        for (int attempt = 1; attempt <= maxRetry; attempt++) {
            try {
                String result = flaky.fetch();
                logger.log("INFO", attempt + "번째 시도 성공: " + result);
                return result;                       // 정상 흐름 복귀 = 복구
            } catch (SQLException e) {
                logger.log("WARN", attempt + "번째 시도 실패: " + e.getMessage());
            }
        }
        logger.log("ERROR", "재시도 " + maxRetry + "회 모두 실패");
        throw new RuntimeException("재시도 " + maxRetry + "회 모두 실패했습니다."); // 통보
    }
}
```

`return`이 `try` 안에 있는 게 포인트예요. 성공하는 순간 메서드를 빠져나가니, 더 이상 재시도하지 않아요. ("catch에서 잡고 그냥 넘어가기"만 하면 블랙홀 — 여기선 성공 시 반환/전부 실패 시 통보로 원칙을 지켜요.)

</details>

**확인**: `FlakyService(2)`(2번 실패 후 성공)를 넘기면 로그가 `WARN, WARN, INFO`로 남고 `데이터-OK`가 반환되면 성공이에요. `FlakyService(99)`를 넘기면 `WARN×3, ERROR` 후 예외가 던져지면 성공이에요.

### Step 3. 예외 회피 — 호출자에게 넘기기 (`DataService.java`)

**목표**: 자신이 처리하지 않고, **호출한 쪽으로 예외를 넘기는** 두 가지 방식을 익혀요.

**할 일**
1. `throws`로 그대로 넘기는 메서드를 만들어요. (예: `void avoidByThrows(FlakyService f) throws SQLException`)
2. 잡아서 **로그만 남기고 다시 던지는** 메서드를 만들어요. (`throw e`로 원래 예외 그대로)

<details>
<summary>💡 힌트 보기</summary>

```java
// (a) throws로 그대로 넘김 — 처리 책임을 호출자에게 위임
void avoidByThrows(FlakyService f) throws SQLException {
    f.fetch();
}

// (b) 로그 등 부가 작업만 하고, 처리는 다시 던져 호출자에게 맡김
void avoidByRethrow(FlakyService f) throws SQLException {
    try {
        f.fetch();
    } catch (SQLException e) {
        logger.log("WARN", "회피: 여기서 처리하지 않고 호출자에게 넘김 - " + e.getMessage());
        throw e;   // 원래 예외를 그대로 다시 던진다
    }
}
```

회피는 **"이 예외는 나보다 호출한 쪽이 처리하는 게 맞다"는 분명한 이유**가 있을 때만 써요. 아무 데서나 `throws`로 떠넘기는 건 안티패턴이에요.

</details>

**확인**: 두 메서드 모두, 예외가 발생하면 **자신은 끝내지 않고 호출자에게 전달**되면 성공이에요. (b)는 던지기 전에 로그가 한 줄 남아요.

### Step 4. 예외 전환 — 의미 있는 예외로 바꾸기 (`DataService.java`)

**목표**: 저수준 `SQLException`을, 호출자가 알아듣는 **업무적 의미의 예외**로 바꿔 던져요. **원인은 꼭 보존**해요.

**할 일**
1. `RuntimeException`을 상속한 `DuplicateUserIdException`을 만들고, **원인(cause)을 받는 생성자**를 둬요.
2. `registerUser(id)`에서 `insertUser(id)`(중복 시 `SQLException`, SQLState `23000`)를 호출해요.
3. 중복(`23000`)이면 `ERROR` 로그 후 `DuplicateUserIdException`으로 **전환해서 던져요**(원인 e 포함).

<details>
<summary>💡 힌트 보기</summary>

```java
void registerUser(String id) {
    try {
        insertUser(id);
    } catch (SQLException e) {
        if ("23000".equals(e.getSQLState())) {           // 무결성 제약 위반 = 중복
            logger.log("ERROR", "아이디 중복: " + id);
            throw new DuplicateUserIdException(id, e);    // 의미 부여 + 원인 보존
        }
        logger.log("ERROR", "회원 저장 중 DB 오류: " + id);
        throw new RuntimeException("회원 저장 중 DB 오류", e);
    }
}

void insertUser(String id) throws SQLException {
    throw new SQLException("Duplicate entry", "23000");  // 시연용
}

static class DuplicateUserIdException extends RuntimeException {
    DuplicateUserIdException(String id, Throwable cause) {
        super("이미 존재하는 아이디입니다: " + id, cause);   // cause로 원인 보존
    }
}
```

`super(메시지, cause)`로 원인을 넘기는 게 핵심이에요. 그래야 나중에 `e.getCause()`로 원래 `SQLException`을 추적할 수 있어요. (스프링이 `SQLException`을 `DataAccessException`으로 바꿔주는 게 바로 이 방식이에요.)

</details>

**확인**: `registerUser("kim")` 호출 시 `DuplicateUserIdException`이 던져지고, 잡아서 `getCause()`를 보면 원래 `SQLException`이 들어 있으면 성공이에요.

### Step 5. 마무리 — Main에서 실행하고 바탕화면 로그 확인 (`Main.java`)

**목표**: 세 전략을 실행하고, 바탕화면에 로그가 **누적**되는 걸 직접 확인해요.

**할 일**
1. `FileLogger`와 `DataService`를 만들어요.
2. (복구) `FlakyService(2)`로 재시도 성공, `FlakyService(99)`로 전부 실패를 각각 실행해요. 실패는 `try/catch`로 받아 메시지를 출력해요.
3. (전환) `registerUser("kim")`을 `try/catch`로 받아 메시지와 `getCause()`를 출력해요.
4. 마지막에 `logger.getLogFilePath()`를 출력하고, **그 파일을 직접 열어** 로그가 쌓였는지 확인해요.

<details>
<summary>💡 힌트 보기</summary>

```java
public class Main {
    public static void main(String[] args) {
        FileLogger logger = new FileLogger();
        DataService service = new DataService(logger);

        try {
            System.out.println("최종 결과: " + service.fetchWithRetry(new FlakyService(2)));
        } catch (RuntimeException e) { System.out.println("실패 통보: " + e.getMessage()); }

        try {
            service.fetchWithRetry(new FlakyService(99));
        } catch (RuntimeException e) { System.out.println("실패 통보: " + e.getMessage()); }

        try {
            service.registerUser("kim");
        } catch (DataService.DuplicateUserIdException e) {
            System.out.println("잡힘: " + e.getMessage());
            System.out.println("원인 보존: " + e.getCause());
        }

        System.out.println("로그 파일: " + logger.getLogFilePath());
    }
}
```

프로그램을 **두 번 실행**한 뒤 로그 파일을 열어보세요. 지워지지 않고 **계속 쌓이면** append가 잘 된 거예요.

</details>
